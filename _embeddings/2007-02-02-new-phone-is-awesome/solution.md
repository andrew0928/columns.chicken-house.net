## Case #1: 多日待機與電力管理實作

### Problem Statement（問題陳述）
**業務場景**：使用者每日通話三至四通，早上以 MP3 作鬧鐘，平日需要手機可連續使用四到五天，避免出差或週末頻繁充電。舊機待機僅一天且半電即斷訊，造成通聯不穩與焦慮。新入手 Dopod C720w 官方標示待機 250 小時，期待能滿足長續航需求。  
**技術挑戰**：在維持藍牙/同步與鬧鐘使用的前提下，實現多日待機並避免低電壓時的基頻不穩。  
**影響範圍**：電話漏接、通訊中斷、行程延誤、使用體驗下降。  
**複雜度評級**：低

### Root Cause Analysis（根因分析）
**直接原因**：
1. 舊機電池老化與電源管理不佳，實際容量衰退導致續航不足。  
2. 韌體在低電壓時的射頻穩定度差，半電時仍可能斷訊。  
3. 背景同步/無線常開增加待機耗電。  

**深層原因**：
- 架構層面：SoC 與射頻模組省電策略不足（例如 DRX/空閒態管理不完善）。  
- 技術層面：OS 缺乏動態電源管理策略與細緻的無線電節能配置。  
- 流程層面：未建立週期性電池校正與耗電行為檢視流程。

### Solution Design（解決方案設計）
**解決策略**：採用續航更佳且電源管理成熟的機型（C720w），配合電池校正、無線節電與同步時段化設定，在不犧牲日常功能的情況下達成多日續航。

**實施步驟**：
1. 電池校正與健康檢測  
- 實作細節：滿充→用到關機→關機狀態下滿充一次，建立 SOC 基準。  
- 所需資源：原廠充電器/USB 充電、電量監測工具。  
- 預估時間：0.5 天（含循環）。

2. 無線與同步策略化  
- 實作細節：關閉閒置藍牙/掃描；ActiveSync 設定尖峰/離峰同步時段；關閉不必要的自動拉取。  
- 所需資源：ActiveSync、Exchange 設定。  
- 預估時間：30 分鐘。

3. 顯示與背光調校  
- 實作細節：自動背光、縮短螢幕逾時；降低待機時 CPU 負載。  
- 所需資源：系統設定、可能的省電工具。  
- 預估時間：20 分鐘。

**關鍵程式碼/設定**：
```ini
[設定路徑]
開始 > 設定 > 連線 > Bluetooth > 關閉自動探索
開始 > 設定 > 行事曆/電子郵件同步 > 尖峰時段每30~60分/離峰手動
開始 > 設定 > 螢幕 > 背光 自動 / 逾時30秒
```

實際案例：文章敘述「一天講三四通、早上當 MP3 鬧鐘，可連續用四五天」，官方待機 250hr。  
實作環境：Dopod C720w（Windows Mobile 5 Smartphone），ActiveSync 4.x，Exchange Server（如 2003 SP2/2007）。  
實測數據：  
- 改善前：待機 1 天，半電易斷訊。  
- 改善後：連續使用 4–5 天（官標 250hr 待機）。  
- 改善幅度：續航約提升 300–400%（以 1→4–5 天估算）。

Learning Points（學習要點）
核心知識點：  
- 行動裝置電源管理策略（無線節能、背光、同步頻率）  
- 電池校正對 SOC 估算的影響  
- 使用情境對續航的關鍵因子

技能要求：  
- 必備技能：Windows Mobile 基本設定、ActiveSync 基礎  
- 進階技能：耗電分析與韌體設定微調

延伸思考：  
- 應用於任何行動裝置的省電優化  
- 可能受限於電池老化與無法調校的韌體策略  
- 可進一步以記錄耗電曲線做數據化優化

Practice Exercise（練習題）  
- 基礎練習：完成尖峰/離峰同步與背光調校。  
- 進階練習：記錄 3 天耗電曲線，分析哪項設定影響最大。  
- 專案練習：建立一份「行動裝置省電 SOP」並附帶量化結果。

Assessment Criteria（評估標準）  
- 功能完整性（40%）：續航明顯提升且不影響日常使用  
- 程式碼品質（30%）：設定文件完整、可重現  
- 效能優化（20%）：耗電下降可量化  
- 創新性（10%）：提出額外節能手法（如自動情境切換）


## Case #2: 系統穩定性與當機防治

### Problem Statement（問題陳述）
**業務場景**：過去使用的手機兩年多來常當機，影響通話與工作安排。新機 C720w 連續半月使用「終於不會當機」，期望建立可重現的穩定策略。  
**技術挑戰**：在安裝多款應用、長時間待機與通話下，維持系統穩定與記憶體健康，避免軟硬體衝突。  
**影響範圍**：通聯中斷、資料遺失、信任感下降。  
**複雜度評級**：中

### Root Cause Analysis（根因分析）
**直接原因**：
1. 舊機 OS/韌體穩定性不足，資源回收有缺。  
2. 記憶體/儲存壓力大，導致系統資源耗盡。  
3. 第三方 App 相容性差，發生死鎖或崩潰。  

**深層原因**：  
- 架構層面：行動 OS 記憶體保護與例外隔離不足。  
- 技術層面：應用未遵循 WM 記憶體/生命週期最佳實踐。  
- 流程層面：缺乏安裝前驗證與回溯機制。

### Solution Design（解決方案設計）
**解決策略**：在更成熟的 OS 與韌體基礎上，建立安裝白名單、記憶體門檻監控與備援策略，確保核心通聯功能優先。

**實施步驟**：
1. 建立應用白名單  
- 實作細節：只安裝穩定度佳之鬧鐘/鍵盤鎖/時鐘等。  
- 所需資源：社群評測清單。  
- 預估時間：1 小時。

2. 記憶體健康與儲存監控  
- 實作細節：常態保持 >20MB 程式記憶體與 >30MB 儲存空間。  
- 所需資源：系統監控工具。  
- 預估時間：30 分鐘設定。

3. 定期備份與回復流程  
- 實作細節：ActiveSync 完整備份；安裝前建立還原點（備份 PIM/檔案）。  
- 所需資源：ActiveSync、備份工具。  
- 預估時間：30 分鐘。

**關鍵程式碼/設定**：
```ini
[健康門檻]
程式記憶體 > 20MB
儲存空間  > 30MB
[備份]
ActiveSync > 工具 > 備份/還原（每週一次）
```

實際案例：半個多月使用未當機（主觀描述）。  
實作環境：C720w（WM5），常駐基本工具 App。  
實測數據：  
- 改善前：經常當機（未量化）。  
- 改善後：0 當機/半月（主觀）。  
- 改善幅度：顯著提升穩定性（質性）。

Learning Points  
- 核心知識點：行動 OS 穩定性要素、白名單策略、備援  
- 必備技能：ActiveSync 備份、檔案管理  
- 進階技能：故障隔離、相容性測試

延伸思考  
- 可應用於任何嵌入式裝置管理  
- 限制：主觀測量、樣本時間有限  
- 優化：導入自動健康檢查腳本

Practice Exercise  
- 基礎：建立備份並清理空間  
- 進階：設計安裝/回溯 SOP  
- 專案：制定團隊行動裝置穩定性準則

Assessment Criteria  
- 功能完整性：白名單+備份+門檻齊備  
- 程式碼品質：SOP 清楚、可重現  
- 效能優化：穩定性可觀察提升  
- 創新性：提出自動監控手段


## Case #3: 橫向 QVGA（320x240）與可讀性最佳化

### Problem Statement（問題陳述）
**業務場景**：需要在手機上長時間閱讀聯絡人、行程、RSS 與文件，過去小螢幕低解析造成閱讀疲勞。C720w 採用橫向 QVGA 顯示，期待提升閱讀效率與舒適度。  
**技術挑戰**：在有限像素密度下，透過字體、主題與對比度調整，讓 UI/內容達到最佳可讀性。  
**影響範圍**：閱讀速度、眼睛疲勞、操作失誤率。  
**複雜度評級**：低

### Root Cause Analysis（根因分析）
**直接原因**：
1. 舊機解析度/亮度不足。  
2. 字體渲染與對比度設定不佳。  
3. 應用未針對橫向介面優化。  

**深層原因**：  
- 架構層面：UI 樣式與佈局未自適應密度/方向。  
- 技術層面：缺乏 ClearType/抗鋸齒與高對比主題。  
- 流程層面：未建立顯示最佳化準則。

### Solution Design（解決方案設計）
**解決策略**：採用較高可視區域的橫向顯示，並通過字體、主題、對比與亮度調整，建立閱讀友善配置。

**實施步驟**：
1. 字體與主題調整  
- 實作細節：選用清晰字體、深淺對比高的主題。  
- 所需資源：主題設定、字體包（如內建）。  
- 預估時間：15 分鐘。

2. 背光與對比度  
- 實作細節：室內降低亮度、室外提高；避免炫光。  
- 所需資源：系統設定。  
- 預估時間：10 分鐘。

3. App 檢視優化  
- 實作細節：RSS/文件查看切換全螢幕模式、調整字級。  
- 所需資源：應用內設定。  
- 預估時間：20 分鐘。

**關鍵程式碼/設定**：
```ini
[顯示設定]
開始 > 設定 > 螢幕 > 字體大小：中/大
開始 > 設定 > 螢幕 > 背光：自動
各 App：啟用全螢幕/閱讀模式
```

實際案例：使用者回饋「LCD 畫面真棒」。  
實作環境：C720w（320x240 橫向 QVGA）。  
實測數據：  
- 改善前：閱讀吃力（質性）。  
- 改善後：閱讀舒適（質性）。  
- 改善幅度：主觀體感顯著提升。

Learning Points  
- 字體渲染與主題對可讀性的影響  
- 橫向佈局對資訊密度的優勢  
- 背光對眼睛疲勞的影響

技能要求  
- 必備：顯示/主題設定  
- 進階：UI 可讀性評估

延伸思考

## Case #4: Outlook 聯絡人與相片同步（含藍牙）

### Problem Statement（問題陳述）
**業務場景**：聯絡人資料需與桌面 Outlook 維持一致，含大頭照；過去手動維護耗時且易錯。C720w 與 Outlook 整合度高，藍牙一鍵同步即可完成。  
**技術挑戰**：在無線環境下正確建立 ActiveSync over Bluetooth，並確保聯絡人欄位與照片映射正確。  
**影響範圍**：通訊效率、撥號準確性與 CRM 記錄品質。  
**複雜度評級**：中

### Root Cause Analysis（根因分析）
**直接原因**：
1. 過去無同步機制或流程複雜。  
2. 無線配對與序列埠設定不當。  
3. 聯絡人欄位不一致導致同步遺漏。  

**深層原因**：  
- 架構層面：缺少一致性的 PIM（Personal Information Manager）資料模型。  
- 技術層面：藍牙 ActiveSync 虛擬 COM 映射易配置錯誤。  
- 流程層面：缺乏定期同步與備份流程。

### Solution Design（解決方案設計）
**解決策略**：設置 Bluetooth ActiveSync，校驗聯絡人欄位映射，建立每日至少一次的自動同步與備份節奏。

**實施步驟**：
1. 藍牙配對與 ActiveSync 服務  
- 實作細節：PC 端開啟「允許裝置連線」，建立 COM 埠；手機選擇「ActiveSync 服務」。  
- 所需資源：ActiveSync 4.x、藍牙堆疊。  
- 預估時間：20 分鐘。

2. 聯絡人欄位與相片  
- 實作細節：在 Outlook 填寫相片，測試同步；若欄位映射異常，清除快取重試。  
- 所需資源：Outlook 2003/2007。  
- 預估時間：20 分鐘。

3. 例行同步與備份  
- 實作細節：每日上下班各同步一次；每週完整備份。  
- 所需資源：ActiveSync。  
- 預估時間：10 分鐘。

**關鍵程式碼/設定**：
```ini
[PC 端]
ActiveSync > 檔案 > 連線設定 > 允許藍牙裝置連線 (COMx)
[手機端]
ActiveSync > 功能表 > 連線透過 Bluetooth > 選擇 PC
Outlook > 連絡人 > 新增/更新相片後觸發同步
```

實際案例：藍芽一按資料全同步；Outlook 相片自動帶入手機。  
實作環境：C720w（WM5）、ActiveSync 4.x、Outlook 2003/2007。  
實測數據：  
- 改善前：手動維護易錯。  
- 改善後：一鍵同步（秒～分鐘級）。  
- 改善幅度：同步成本大幅下降（質性）。

Learning Points  
- ActiveSync over Bluetooth 流程  
- PIM 欄位映射注意事項  
- 例行同步與備份設計

技能要求  
- 必備：藍牙配對、ActiveSync 基礎  
- 進階：PIM 欄位對齊、故障排除

延伸思考  
- 可擴展至行事曆、工作與郵件同步  
- 風險：藍牙安全性與洩露  
- 優化：改用 USB 初始全量，同步差異以藍牙

Practice Exercise  
- 基礎：完成藍牙 ActiveSync 配對  
- 進階：含相片的聯絡人完整往返同步  
- 專案：編寫同步 SOP 與還原流程

Assessment Criteria  
- 功能完整性：聯絡人+相片同步成功  
- 程式碼品質：設定清楚可重現  
- 效能優化：同步時間縮短  
- 創新性：安全性強化措施

## Case #20: Outlook 聯絡人照片自動帶入手機

### Problem Statement
**業務場景**：希望來電時顯示聯絡人大頭照，以利快速辨識；需從桌面 Outlook 自動同步至手機。  
**技術挑戰**：確保照片尺寸/格式符合行動端要求，避免同步失敗或顯示變形。  
**影響範圍**：辨識速度、接聽決策、來電體驗。  
**複雜度評級**：低

### Root Cause Analysis
**直接原因**：
1. 電話端無法方便維護照片。  
2. 照片過大導致傳輸/顯示問題。  
3. 欄位映射不一致。  

**深層原因**：  
- 架構：PIM 圖像欄位與封裝差異。  
- 技術：壓縮格式與解析度限制。  
- 流程：缺乏統一的照片維護流程。

### Solution Design
**解決策略**：在 Outlook 端統一照片尺寸（如 96x96/128x128）、JPG 格式，透過 ActiveSync 自動同步，驗證來電顯示。

**實施步驟**：
1. 照片處理  
- 實作細節：批次縮圖至 96–128px 正方形。  
- 資源：影像處理工具。  
- 時間：30–60 分鐘。

2. 聯絡人更新與同步  
- 實作細節：將照片貼至 Outlook 聯絡人，執行同步。  
- 資源：Outlook、ActiveSync。  
- 時間：20 分鐘。

**關鍵程式碼/設定**：
```powershell
# 批次縮圖（示意，需 ImageMagick）
magick *.jpg -resize 128x128^> thumbs\%~n.jpg
```

實際案例：Outlook 裡的大頭照自動進來手機。  
實作環境：Outlook 2003/2007、ActiveSync、C720w。  
實測數據：  
- 改善前：來電無照片或錯亂。  
- 改善後：來電顯示清晰。  
- 改善幅度：辨識體驗提升。

Learning Points  
- 聯絡人照片欄位與限制  
- 影像尺寸與壓縮  
- 批次處理流程

技能要求  
- 必備：Outlook 聯絡人編輯  
- 進階：批次影像處理

延伸思考  
- 與 AD/Exchange 照片欄位整合  
- 風險：肖像權與隱私  
- 優化：自動化照片同步

Practice Exercise  
- 基礎：為 10 位聯絡人加照片  
- 進階：批次縮圖與同步  
- 專案：建立公司花名冊同步流程

Assessment Criteria  
- 功能完整性：來電顯示正確  
- 程式碼品質：批處理腳本  
- 效能優化：照片大小合理  
- 創新性：自動化整合
