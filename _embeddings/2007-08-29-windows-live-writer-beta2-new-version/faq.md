# Windows Live Writer Beta 2（新版）FAQ

## 問題與答案 (FAQ)

### Q&A 類別 A: 概念理解類

Q1: 什麼是 Windows Live Writer（WLW）？
- A簡: 微軟推出的桌面部落格編輯器，支援離線撰寫與所見即所得預覽，對多平台相容。
- A詳: Windows Live Writer 是微軟推出的免費桌面部落格編輯器，主要用於撰寫、預覽與發佈部落格文章。它提供所見即所得（WYSIWYG）編輯體驗、離線草稿管理、圖片與媒體插入、分類與標籤設定等，並透過標準的部落格 API（如 MetaWeblog XML-RPC、AtomPub）與多數平台（WordPress、Blogger、Movable Type、TypePad、Windows Live Spaces 等）相容。WLW 的核心優勢是「真實預覽」：可下載部落格佈景主題與 CSS，在本機模擬實際樣式，降低發佈後版面跑掉的風險。對於需要穩定編輯、不想侷限瀏覽器後台、或常在離線環境撰寫的使用者，WLW 是高效率的解決方案。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: A-Q4, A-Q5, A-Q9, B-Q1

Q2: 什麼是 Windows Live Writer Beta 2？
- A簡: WLW 的第二個測試版，介面優化、相容性提升，改善輸入法問題。
- A詳: Windows Live Writer Beta 2 是 WLW 的第二個公開測試版本，於 2007/05/30 釋出。相較先前測試版，Beta 2 在使用者介面（UI）上有明顯改進，操作一致性與易用性提高，並改善了某些輸入法（IME）在特定情況下無法啟用的問題。此外，Beta 2 提升與常見部落格平台的相容性與自動設定流程的穩定性，讓新增帳號與真實預覽更可靠。作為 Beta 軟體，它仍可能存在錯誤或功能調整，但已足以日常使用，特別是與 WordPress 或 Blogger 搭配。對於希望體驗新版功能並提供回饋的使用者，Beta 2 是值得嘗試的節點版本。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: A-Q3, A-Q8, D-Q6

Q4: 為什麼需要桌面部落格編輯器而非純瀏覽器後台？
- A簡: 提供離線撰寫、穩定編輯與真實預覽，降低版面與連線依賴。
- A詳: 桌面編輯器的價值在於提升撰寫體驗與可靠性。與瀏覽器後台相比，WLW 支援離線草稿，無網路也能編輯；其本機所見即所得預覽可下載佈景與 CSS，接近發佈結果，避免多次「發佈—修排」。在長文或含大量圖片的文章中，桌面應用程式的穩定性與快捷鍵體驗通常優於瀏覽器。它也整合圖片處理（縮放、外框、水印）與分類標籤管理，節省在多個工具間切換的成本。對於常在移動中、網路不穩、或需要批次處理文章的作者，WLW 的桌面化流程顯著降低編輯摩擦。
- 難度: 初級
- 學習階段: 基礎

Q5: Windows Live Writer 的核心價值是什麼？
- A簡: 真實預覽、離線草稿、多平台相容與可擴充外掛生態。
- A詳: WLW 的核心價值可歸納為四點。第一，真實預覽：透過下載佈景與 CSS，本機渲染接近上線樣式，發文前就能校正版面。第二，離線草稿：本機儲存，隨時撰寫，網路恢復後再發佈或同步至遠端草稿。第三，相容性：支援主流平台與標準 API（MetaWeblog、AtomPub、Blogger API），多部落格帳號切換便利。第四，可擴充：外掛架構允許開發者新增功能，如語法高亮、SEO 輔助、圖片批次處理等。綜合上述，WLW 將寫作、排版、媒體處理與發佈統整於單一工作流，為頻繁創作的部落客與內容團隊提供高效、穩定且可擴充的工具選擇。
- 難度: 初級
- 學習階段: 核心

Q6: WLW 與瀏覽器後台編輯的差異是什麼？
- A簡: WLW 提供本機所見即所得、離線草稿與圖片處理，瀏覽器較仰賴網路與外掛。
- A詳: 瀏覽器後台的優點是零安裝、直接即用，但高度仰賴網路、瀏覽器相容性與站台的 JS/CSS。WLW 作為桌面應用，能提供更穩定的編輯體驗、完整快捷鍵、與一致的圖片與版面控制。它將標籤、分類、草稿同步與圖片上傳整合在單一工具中，減少「切工具」與貼上格式錯亂的問題。WLW 的真實預覽能模擬部落格樣式；反之瀏覽器後台若未支援即時預覽，需不斷預覽/發佈驗證。缺點是 WLW 須安裝、維護版本相容，且某些平台新功能需等待支援。總結：重視穩定與效率者偏好 WLW，輕量與一次性發文則瀏覽器更直接。

Q7: Beta 版與正式版的差異與風險是什麼？
- A簡: Beta 功能未定且可能不穩定；正式版較穩定並提供完整支援。
- A詳: Beta 版（如 Beta 2）屬於測試階段，目的在收集回饋、驗證功能與相容性。它可能包含未完成的功能、臨時性介面、記憶體或相容性問題，更新頻繁且變動較大。正式版（RTM）則經過完整測試與品質把關，文件、支援與相容性較完整，較適合生產環境。採用 Beta 的風險包括：資料丟失、外掛不相容、預覽誤差或 API 行為變動。最佳實踐：安裝前備份 WLW 設定與草稿、在非關鍵主機先行測試、確認部落格平台版本與外掛相容情形、必要時保留回退機制。

Q8: Beta 2 與 Beta 1 的主要差異有哪些？
- A簡: Beta 2 改善 UI 可用性、IME 叫不出來等問題，並提升平台相容。
- A詳: 相較 Beta 1，Beta 2 著重在易用性與相容性：介面更新讓工具列與常用操作更直觀，降低設定門檻；針對部分語系輸入法（IME）在特定焦點/控制元件下無法啟動的問題進行修正，提升中文、日文等用戶的編輯可靠度；自動偵測部落格設定（如 RSD 探測）更穩定，與 WordPress/Blogger 等平台的 API 互動更順暢；此外，圖片處理與預覽管線也有優化，減少發佈後樣式落差。整體來看，Beta 2 更貼近可日用的版本，雖仍可能有邊角問題，但已能與多數部落格「搭起來」運作。

Q9: WLW 支援哪些常見部落格平台？
- A簡: 支援 WordPress、Blogger、Movable Type、TypePad、Windows Live Spaces 等。
- A詳: WLW 透過標準 API 與多數平台相容。以 WordPress 為例，使用 MetaWeblog XML-RPC（/xmlrpc.php）進行文章、分類、媒體操作；Blogger 則透過 Google 的 Blogger API 或 AtomPub；Movable Type 與 TypePad 同樣支援 MetaWeblog/MT API；Windows Live Spaces 有內建整合。多平台相容的好處是可在單一工具管理多個網站，切換帳號快速，草稿格式與媒體處理一致。注意：不同平台與版本的 API 能力與權限略有差異（如標籤支援、頁面與文章的操作範圍），設定時需選擇正確平台類型與端點。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q1, C-Q2, D-Q1

Q10: WLW 如何與部落格平台達成相容？
- A簡: 透過標準 API（MetaWeblog、AtomPub）與 RSD 自動探索設定。
- A詳: 相容性主要依賴通用協定與自動設定。WLW 會在新增帳號時嘗試解析站台首頁的 RSD（Really Simple Discovery）連結，以找出 API 端點與支援能力；若無法自動發現，則提供手動輸入端點（如 WordPress 的 /xmlrpc.php）。發佈時，WLW 將文章內容、分類與媒體透過 XML-RPC 或 AtomPub 組成請求，並進行驗證（基本認證或平台特定流程）。成功後，伺服器回傳文章 ID 與狀態，WLW 更新本機映射以追蹤後續編輯。這種以協定為核心的設計，讓 WLW 能跨平台運作，同時保留平台特性（如自訂欄位）透過擴充實現。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q1, B-Q7, C-Q2

Q11: 什麼是 WLW 的「真實預覽」？
- A簡: 下載部落格佈景與 CSS，在本機渲染接近上線的樣式。
- A詳: 真實預覽（Theme-aware Preview）是 WLW 的關鍵能力。新增帳號後，WLW 嘗試抓取部落格的主題模板與 CSS，建立本機預覽頁面框架。在編輯時，內容會被注入這個框架中渲染，讓使用者看到接近發佈後的排版、字體與顏色。此機制減少來回預覽與修改的時間，尤其對複雜樣式、圖片多的文章很有幫助。若主題使用大量客製 JS/CSS 或需伺服器端渲染的區塊，預覽可能與實際略有差異，此時可透過「下載最新佈景」、「停用特定腳本」或「切換純樣式預覽」來校正。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q3, C-Q4, D-Q2

Q12: 什麼是輸入法（IME），為何影響 WLW 使用？
- A簡: IME 是中文等語系輸入工具；早期版本有焦點觸發問題。
- A詳: IME（Input Method Editor）是中文、日文、韓文等語系用戶輸入文字的必要元件。部分 Windows 文本控制項在處理焦點、訊息迴圈或 TSF/IMM32 框架時，若實作不完整，可能導致 IME 無法被正確喚起或候選窗不顯示。WLW 在早期測試版本上，曾出現特定欄位（例如標題或內文編輯器）無法啟動 IME 的狀況，造成輸入受阻。Beta 2 針對此問題做了修正，讓 IME 在大多數場景正常運作。若仍遇到問題，可嘗試更新系統 IME、切換輸入法或啟用相容模式，並回報具體情境以利後續改善。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: D-Q6, B-Q4

Q15: WLW 與 Microsoft Word 撰寫後貼上的差異？
- A簡: WLW 產生乾淨的網頁格式與真實預覽；Word 貼上常帶冗餘樣式。
- A詳: 從 Word 貼上到後台常出現冗餘標記、內聯樣式與奇怪段落分隔，導致前台樣式不一致。WLW 以網頁內容為核心，提供乾淨的 HTML 與針對網頁的段落/清單控制，並可在本機預覽真實樣式，減少來回校正。它也整合了圖片處理與標籤分類，讓整體工作流更順暢。若需要從 Word 移轉內容，建議先以「貼上為純文字」或使用 WLW 的格式清理功能，再在 WLW 中重建標題、清單與圖片樣式。
- 難度: 初級
- 學習階段: 核心
- 關聯概念: C-Q5, A-Q11

Q16: 什麼是 RSD（Really Simple Discovery）？
- A簡: 一種在網站上宣告部落格 API 端點的自動探索機制。
- A詳: RSD 是一種簡單的 XML 描述檔，由網站在首頁 HTML 以 link rel 宣告，指出可用的遠端發佈 API 端點（如 MetaWeblog、AtomPub）與服務資訊。WLW 在新增帳號時會嘗試抓取並解析 RSD，以自動填入 API 位址與能力，減少使用者手動設定與出錯的機會。若站台未提供 RSD，WLW 仍允許手動指定端點。理解 RSD 有助於診斷「自動設定失敗」或「端點錯誤」等問題，並協助網站管理員正確配置跨工具的相容性。
- 難度: 中級
- 學習階段: 進階
- 關聯概念: B-Q7, C-Q2, D-Q1

Q17: WLW 的外掛（Plugin/Add-in）是什麼？
- A簡: 以擴充編輯/發佈功能的外部元件，通常基於 .NET 實作。
- A詳: WLW 支援外掛架構，允許第三方開發者擴充功能，例如程式碼區塊語法高亮、SEO 標題建議、圖片批次水印、短網址生成、範本片段插入等。外掛通常以 .NET（C# 或 VB）撰寫，透過 WLW 的 SDK 介面與事件掛入編輯器或發佈流程。這讓 WLW 在維持核心輕量的同時，滿足進階使用者或專業工作流需求。安裝外掛需注意版本相容性與安全性，來源不明的外掛可能影響穩定或資料安全。
- 難度: 中級
- 學習階段: 進階
- 關聯概念: B-Q6, C-Q10

Q18: 什麼是離線草稿與同步？
- A簡: 本機儲存文章，離線撰寫；線上時可發佈或同步為遠端草稿。
- A詳: 離線草稿是 WLW 的核心工作流之一。使用者可在無網環境撰寫並儲存至本機，之後上線時選擇「發佈」或「上傳為草稿」至平台後台。WLW 會維持本機與遠端的映射（ID 與版本），支持後續雙向更新。在多人協作或跨裝置情境，建議將關鍵草稿上傳為遠端草稿，避免本機損壞或遺失風險。若平台支援版本控制或修訂（如 WordPress），可搭配提高回溯安全度。
- 難度: 初級
- 學習階段: 核心
- 關聯概念: B-Q8, C-Q8

Q19: WLW 對圖片媒體處理有何特點？
- A簡: 內建插圖、縮放、外框與上傳，支援多種上傳目標。
- A詳: WLW 提供圖片插入、即時縮放、裁切、外框、陰影與水印等常用編輯，並能在發佈時自動上傳圖片至部落格媒體庫或指定上傳目的地（如 FTP）。它會在內容中產生正確的引用路徑，並可設定預設尺寸與對齊方式。對於圖片密集的文章，這降低了出文前的圖像準備成本。注意：若平台的媒體上傳權限或路徑設定不標準，可能需手動調整端點或憑證。

Q20: 為何作者說「跟我的 blog 搭起來還是可以用」？

Q1: WLW 與部落格服務的連線與驗證流程如何運作？

Q2: WLW 的發佈流程背後機制是什麼？

- A詳: 技術原理說明：編輯器產生文章結構（標題、內文 HTML、分類/標籤、狀態、媒體引用）。傳輸層將其封裝為 XML-RPC（metaWeblog.newPost/editPost）或 AtomPub（POST/PUT）請求。關鍵步驟或流程：1) 圖片檔案先上傳取得 URL；2) 內容中的本機引用替換為遠端 URL；3) 送出文章與中繼資料；4) 接收回應（文章 ID、發佈狀態）；5) 更新本機草稿與遠端文章映射。核心組件介紹：Editor（內容/格式）、Media Manager（上傳與路徑替換）、Publisher（發佈協調）、Mapping Store（ID 對應）。錯誤處理涵蓋重試、回滾與草稿保存。

Q3: WLW 的真實預覽如何實作？

- A詳: 技術原理說明：WLW 嘗試抓取部落格的樣板頁與關聯 CSS，建立一個本機的預覽框架（HTML + CSS），並將使用者內容注入指定容器節點。關鍵步驟或流程：1) 新增帳號時擷取樣式資源；2) 建立預覽快取；3) 編輯時即時套用樣式；4) 支援切換不同檢視（編輯/預覽/HTML）；5) 允許手動重新下載主題。核心組件介紹：Theme Downloader（資源抓取）、Preview Engine（渲染與合成）、Cache Manager（資源快取）、Sanitizer（避免惡意/衝突腳本）。此設計能接近真實，但對需伺服器端渲染或動態腳本的區塊，會以降級策略處理。

Q8: 如何離線撰寫並稍後發佈？

- A詳: 具體實作步驟：1) 在無網環境撰寫，定期以本機草稿儲存；2) 網路可用時連線帳號；3) 選擇「發佈」或「上傳為草稿」；4) 於後台檢視草稿樣式與連結；5) 確認無誤後再正式發佈。關鍵設定：時區、發佈時間排程、媒體上傳策略。注意事項與最佳實踐：長文定期另存副本；跨裝置時使用遠端草稿保持一致；大幅改版主題時，重新下載佈景以確保預覽準確。
- 難度: 初級
- 學習階段: 核心
- 關聯概念: A-Q18, B-Q8

Q9: 如何備份 WLW 設定與草稿？
- A簡: 匯出帳號與設定、備份本機草稿資料夾，保留回退方案。
- A詳: 具體實作步驟：1) 在 WLW 使用匯出/備份功能（若提供），或手動備份設定檔與帳號；2) 尋找本機草稿儲存路徑（使用者設定檔中的 WLW 資料夾），完整複製至安全位置；3) 記錄版本與外掛清單；4) 進行系統層級還原點或映像備份。關鍵設定：權限（確保備份檔可讀取）、敏感資訊（帳密加密）。注意事項與最佳實踐：在升級或安裝 Beta 前先備份；定期測試還原流程；保存至少兩份備份於不同位置。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: A-Q7, D-Q10

Q10: 如何安裝與管理 WLW 外掛？
- A簡: 從可信來源下載外掛，放入指定資料夾或使用安裝程式並重啟 WLW。
- A詳: 具體實作步驟：1) 取得外掛（官方/知名社群開發者）；2) 依說明放至 WLW 外掛目錄或執行安裝程式；3) 重啟 WLW，於外掛管理器啟用/停用；4) 在編輯器確認功能按鈕/面板出現。關鍵設定：版本相容、權限與網路存取（若外掛需連線）。注意事項與最佳實踐：來源可信、避免同類外掛重複造成衝突；更新前備份設定；遇異常時逐一停用以定位問題；定期檢視外掛維護狀態與安全告警。
- 難度: 中級

### Q&A 類別 D: 問題解決類（10題）

Q1: 無法連線到 WordPress 的 xmlrpc.php 怎麼辦？
- A簡: 檢查端點、啟用 XML-RPC、驗證 HTTPS 與憑證，查看伺服器日誌。
- A詳: 問題症狀描述：WLW 新增帳號或發佈時顯示無法連線/404/500。可能原因分析：端點設定錯誤（非 /xmlrpc.php）、主機阻擋 XML-RPC、伺服器重寫規則衝突、HTTPS 憑證問題、基本認證被關閉或安全外掛阻擋。解決步驟：1) 以瀏覽器直連 https://yourblog/xmlrpc.php 確認回應；2) 檢查 WordPress 設定與安全外掛（允許 XML-RPC）；3) 檢視 .htaccess/伺服器重寫；4) 若需，為 WLW 啟用應用程式密碼；5) 於 WLW 手動指定正確端點。預防措施：保留 RSD、維持伺服器更新、監看錯誤日誌，重大改版後重新測試 WLW。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: C-Q2, B-Q7

Q2: 真實預覽樣式錯亂或與前台不一致？
- A簡: 重新抓取主題、清快取、停用動態腳本，改用純樣式預覽比對。
- A詳: 問題症狀描述：WLW 預覽與實際網站字體、間距或版面不同。可能原因分析：主題更新未同步、CDN/快取導致 CSS 不一致、登入狀態注入管理列、動態腳本干擾。解決步驟：1) 重新下載佈景；2) 清除 WLW 預覽快取；3) 關閉動態腳本或使用純樣式預覽；4) 以遠端草稿在前台檢視對照；5) 檢查是否有登入管理列干擾。預防措施：主題改版後立即重新抓取、遵循可預測的 CSS 版本管理、避免將動態元素作為版面關鍵參考。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q3, C-Q4

Q3: 發佈時出現 401/403 驗證或權限錯誤？
- A簡: 檢查帳密/應用密碼、HTTPS 與伺服器認證設定，確認角色權限。
- A詳: 問題症狀描述：WLW 回報未授權（401）或禁止（403）。可能原因分析：帳密錯誤、兩步驗證未配置應用密碼、伺服器阻擋基本認證、角色權限不足（作者/投稿者）、安全外掛或 WAF 攔截。解決步驟：1) 重設密碼或改用應用程式密碼；2) 啟用 HTTPS，確保認證不被攔截；3) 檢查伺服器/外掛設定允許 API 存取；4) 調整使用者角色或核發具發佈權限的帳號；5) 檢查 WAF 規則白名單。預防措施：最小權限原則、定期檢視存取記錄、採用 OAuth/Token 流程（若平台支援）。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q1, C-Q3

Q4: 圖片無法上傳或發佈後不顯示？
- A簡: 檢查上傳端點與權限、MIME/大小限制，確認內容 URL 已替換。
- A詳: 問題症狀描述：發佈成功但圖片 404 或預覽顯示破圖。可能原因分析：上傳失敗（權限/容量/MIME 限制）、FTP 設定錯誤、HTTPS 混合內容、相對路徑未替換。解決步驟：1) 於 WLW 設定中確認上傳目的地與憑證；2) 檢查平台媒體庫權限與限制；3) 測試單張小圖上傳；4) 查看內容 HTML 路徑是否為公開 URL；5) 清瀏覽器快取與 CDN；6) 若使用 FTP，測試連線與路徑對應。預防措施：統一使用 HTTPS、設定合理大小與壓縮、建立固定上傳路徑策略。
- 難度: 初級
- 學習階段: 核心
- 關聯概念: B-Q5, C-Q5

Q5: 中文亂碼或編碼顯示異常怎麼處理？
- A簡: 確保 UTF-8 一致，避免混用編碼，檢查標頭與資料庫設定。
- A詳: 問題症狀描述：發佈後中文顯示為問號或亂碼。可能原因分析：WLW 與平台編碼不一致、HTTP 標頭/Meta charset 錯誤、資料庫非 UTF-8、外掛轉碼。解決步驟：1) 確認網站使用 UTF-8 並統一 HTML meta 與 HTTP 標頭；2) 檢查資料庫與資料表編碼；3) 測試以純文字張貼；4) 停用可疑外掛；5) 於 WLW 切換/確認內容編碼（若可設定）。預防措施：全站 UTF-8 策略、升級舊站編碼、避免從富文本來源直接貼上造成隱藏轉碼。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: C-Q7, A-Q15

Q6: WLW 中輸入法（IME）無法啟動如何修正？
- A簡: 升級至 Beta 2、切換 IME、檢查焦點與相容模式，更新系統。
- A詳: 問題症狀描述：在標題或內文區無法切換至中文輸入。可能原因分析：早期版本焦點/訊息處理缺陷、IME 設定損壞、DPI/多螢幕造成候選窗定位異常。解決步驟：1) 升級至 Beta 2 或更新版；2) 重新安裝/切換 IME；3) 以相容性模式執行；4) 測試以不同解析度與縮放；5) 更新 Windows 與 IME。預防措施：維持系統更新、避免安裝多個輸入法管理工具、保留最少必要外掛以降低干擾。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: A-Q12, B-Q4, C-Q6

Q7: 無法抓到分類/標籤清單怎麼辦？
- A簡: 重新同步能力、檢查 API 權限與版本，必要時手動輸入分類。
- A詳: 問題症狀描述：分類面板空白或僅顯示部分項目。可能原因分析：API 能力不支援或受限、認證角色權限不足、外掛或主題影響 API 回傳、快取過期。解決步驟：1) 重新整理/同步分類；2) 檢查使用者角色權限；3) 檢視伺服器端 API 回傳（開啟除錯日誌）；4) 停用可疑外掛測試；5) 暫時手動輸入分類。預防措施：維持平台/外掛更新、定期清理快取、採最小權限但足夠的發佈角色。
- 難度: 初級
- 學習階段: 核心
- 關聯概念: B-Q1, C-Q7

Q8: WLW 啟動即當機或異常關閉？
- A簡: 以安全模式/清外掛啟動、重建設定、檢查相依元件與事件檢視器。
- A詳: 問題症狀描述：開啟即崩潰或載入外掛時關閉。可能原因分析：外掛衝突、設定檔損壞、.NET/系統相依缺少、顯示驅動問題。解決步驟：1) 嘗試安全模式或以命令列停用外掛；2) 暫移除外掛資料夾；3) 重建/重置 WLW 設定；4) 安裝/修復相依元件（.NET Framework）；5) 於事件檢視器檢查錯誤並定位 DLL；6) 重新安裝 WLW。預防措施：外掛逐一安裝與測試、版本升級前備份、維持系統與驅動更新。
- 難度: 中級
- 學習階段: 進階
- 關聯概念: B-Q6, C-Q10

Q9: 發佈後時間或時區錯誤如何校正？
- A簡: 對齊網站時區、WLW 時間設定，使用排程發佈並校驗伺服器時間。
- A詳: 問題症狀描述：文章顯示時間不正確或排程錯位。可能原因分析：網站時區與 WLW 本機時區不一致、夏令時間偏移、伺服器時間不準。解決步驟：1) 將網站（WordPress 等）時區設定與本機一致；2) 確認 WLW 使用本機正確時區；3) 測試排程發佈一篇短文；4) 檢查伺服器 NTP 同步；5) 若平台以 UTC 儲存，確認呈現層轉換正確。預防措施：固定 NTP 同步、跨區團隊建立時區準則、發佈前以草稿驗證呈現時間。
- 難度: 初級
- 學習階段: 核心
- 關聯概念: C-Q8, B-Q8

Q10: 想回復舊版或解除安裝 WLW，該怎麼做？
- A簡: 先備份設定與草稿，解除安裝後安裝舊版並恢復備份。
- A詳: 問題症狀描述：升級後不相容或不穩定，需回退。可能原因分析：外掛/平台相容性問題、IME/預覽回歸缺陷。解決步驟：1) 匯出帳號與備份草稿；2) 透過系統移除程式解除安裝 WLW；3) 重新啟動後安裝舊版安裝包；4) 還原設定與外掛（建議逐一恢復以排除問題）；5) 測試核心流程。預防措施：升級前建立還原點、保存不同版本安裝包、在非關鍵環境先驗證、維持備份策略。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: C-Q9, A-Q7
