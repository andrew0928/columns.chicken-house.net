# 心得 - Emerging Developer Patterns for the AI Era

## 問題與答案 (FAQ)

### Q&A 類別 A: 概念理解類

Q1: 什麼是 AI-native Git？
- A簡: AI-native Git將版控焦點從程式碼轉往文件、意圖與測試，追蹤生成脈絡與驗證結果，串連artifact管理。
- A詳: AI-native Git是為AI代理生成的開發流程重新設計的版控觀念。傳統Git精細追蹤人手寫的程式碼；在AI時代，程式碼更像產出物(artifacts)，真正的來源是需求文件、指令(prompt)、測試規格。AI-native Git強調記錄意圖、綁定測試、生成與驗證結果，並讓artifact與來源可回溯。它讓版控從「誰改了哪行」轉為「為何、以何意圖、以何測試生成了什麼」，更適合AI主導的軟體工程。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: A-Q2, B-Q1, B-Q2

Q2: 在AI時代，「source」的意義是什麼？
- A簡: source從「原始碼」轉為「人類意圖的第一手產出」：需求文件、prompt、測試，即可驅動AI生成程式。
- A詳: 過去source等同原始碼；AI普及後，能理解需求並生成程式，code成為AI的產出，source則是人類意圖的首要表述，如需求文件、系統說明、開發指令、測試規格。這些可版控、可回溯、可驗證的「source」成為CI/CD的左移起點。把source視為「對機器有意義的第一份人類輸出」，能更好地連結生成、驗證與部署。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: A-Q1, A-Q11, B-Q8

Q3: 為什麼版控重心從程式碼轉向文件與prompt？
- A簡: 因AI生成能力強，程式碼是產出物。人類價值在於意圖與規格，需被追蹤回溯與驗證，版控應聚焦此處。
- A詳: 人類主要改為定義需求、編寫規格、撰寫prompt與測試。這些「意圖與規範」決定AI如何生成與驗證成果。追蹤意圖變化、綁定測試、紀錄模型與代理上下文，才是可審計、可再現的工程核心。將版控重心移到文件/提示可提升可回溯性，降低黑盒風險，並讓交付品質更穩定。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: B-Q1, B-Q2, A-Q11

Q4: 什麼是 vibe coding？
- A簡: 以自然語言與文件描述需求，AI即時生成架構與程式碼，取代固定樣板與大量搜尋貼上。
- A詳: vibe coding是一種AI驅動的生成式開發方式。開發者用需求敘述、文件與少量指令，AI代理即時產生初始專案、架構、介面與實作，並迭代修正。它替代傳統「套模板」與「複製貼上」，提供更個人化與情境化的產出，並可搭配測試與指南提升品質。vibe coding使開發流程左移，讓「設計與規格」成為驅動點。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: A-Q5, B-Q17, C-Q9

Q5: create-react-app等專案範本與vibe coding差異？
- A簡: 範本給固定起手式；vibe coding依需求動態生成，彈性選stack與架構，可更精準貼近情境。
- A詳: 傳統範本提供通用框架與檔案結構，但限制技術選擇與客製化深度。vibe coding讓AI依需求自由組合工具與架構，能快速試不同tech stack、混搭工具、出多版比較，成本極低。此轉變改變生態分布：從少數頭部框架，走向更廣泛的可組合生成，開發者獲得更大自主性。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: A-Q4, B-Q9, C-Q9

Q6: 什麼是 AI-driven interface？
- A簡: 以AI理解使用者意圖，動態合成介面與資訊，讓UI依情境生成與調整並與對話協作。
- A詳: AI-driven interface指由AI代理根據當下意圖、上下文與環境感知，動態決定呈現的UI模組、內容與操作。它讓儀表板等複雜介面不再固定，而是合成需用的元件。搭配聊天與工具呼叫，AI可引導任務完成。此介面結合人機協作，既保留傳統UI長處，也以AI補齊當下需求變化。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: B-Q3, B-Q5, A-Q7

Q7: 傳統儀表板與動態合成儀表板有何差異？
- A簡: 傳統固定且資訊繁雜；動態合成依當下需求生成元件，支援AI代理操作與人機共視。
- A詳: 傳統儀表板強調「一覽無遺」，但使用門檻高、難精準聚焦。動態合成儀表板以AI理解任務，即時挑選/配置小工具(工具化元件)，用Markdown、Mermaid等渲染視覺，並讓AI與人共同使用同一視圖(Agent-ready)。此設計降低搜尋成本，提升決策效率，並更適合複雜情境。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: B-Q5, B-Q6, C-Q3

Q8: 為何AI驅動UI能提升UX？
- A簡: AI以對話與感知把握當下意圖，動態提供最合適資訊與操作，降低偏離與迷航。
- A詳: 傳統UX靠事前研究推測大多數情境，難覆蓋例外。AI驅動UI透過上下文、事件、對話理解使用者當下意圖，合成恰當的介面與引導流程。當系統即時「懂你要什麼」，滿意度提升。雖仍需迭代與監控，隨模型推理進步，AI路線長期將優於統計驅動的固定設計。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: A-Q9, B-Q3, B-Q18

Q9: 設計師「事前設計」與AI「當下理解意圖」差異？
- A簡: 事前設計靠研究預測主流流程；AI即時理解意圖動態生成介面，更覆蓋少數情境。
- A詳: 事前設計強於精緻的元件與標準流程，受限於資源與統計。AI的當下理解可依個人意圖與環境生成操作，涵蓋冷門情境與變異。未來UI將模組化：元件仍需精良設計，流程組合由AI主導。兩者互補，形成更彈性的人機介面生態。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: B-Q3, A-Q6, C-Q4

Q10: 什麼是 Context Engineering？
- A簡: 系統性管理模型上下文：文件化、檔案化、檢索與載入，讓AI在對的時機獲取對的資訊。
- A詳: Context Engineering是從Prompt Engineering演進而來，重點在「資訊供給」。藉由指令文件、知識庫、任務清單(dev-notes、tasks.md)、檔案系統與MCP工具，將長期內容外置，必要時載入至context window。它如「虛擬記憶體」：寫回、載入、分段、標註，讓AI持續在可控上下文中推理與執行。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q7, B-Q13, C-Q5

Q11: 文件在AI時代扮演什麼角色？
- A簡: 文件成為AI理解與執行的主要上下文來源：規格、索引、互動知識庫與工具載具。
- A詳: 文件不僅給人看，也給AI看。需求、規範、設計、測試、指南以Markdown等格式版控與檔案化，AI以MCP載入/寫回，形成互動知識基。文件是生成程式的驅動、評估的依據、長期記憶的容器，也是團隊協作的真實單一來源，促成軟體工程全面左移。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: A-Q12, B-Q8, C-Q10

Q12: 什麼是「document as code」？
- A簡: 將文件像程式碼管理：用IDE編輯、Git版控、CI/CD發佈，融入開發工具鏈。
- A詳: document as code以程式化方式處理文檔：使用IDE(如VSCode)、Markdown格式、Git版控、PR審閱、CI/CD發佈(靜態站)。好處是流程一致、回溯容易、與程式碼緊密連動。配合AI，文件即成生成與驗證的驅動，形成1+1>2的綜效，提升工程品質與速度。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: B-Q8, C-Q10, A-Q11

Q13: 為何Markdown成為首選文件格式？
- A簡: 可讀可寫、易版控、工具支援廣泛，LLM能生成與理解，並可嵌Mermaid等語法。
- A詳: Markdown兼顧簡潔與結構，適合人類閱讀與機器解析。它天然適配Git與PR、能嵌表格/程式碼/圖表(Mermaid/SVG)、易轉靜態站。LLM生成與解析Markdown成熟，使其成為互動文件、知識庫與動態合成介面的重要承載。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: B-Q22, C-Q3, C-Q10

Q14: API與MCP有何差異？
- A簡: API連SaaS系統；MCP連AI工具。MCP標準化工具暴露與上下文，支援代理推理與行動。
- A詳: API面向服務整合，描述資源與操作；MCP面向AI代理，標準化工具能力、輸入輸出、權限與上下文。MCP讓代理可檢索、讀寫檔案、調用服務、執行動作，並對接OAuth等安全。它是「AI的手腳」標準，延伸API生態到代理世界。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q11, C-Q8, A-Q15

Q15: 為何MCP邁向通用標準？
- A簡: AI代理需統一工具協定以安全可控方式行動。MCP填補代理與工具間的標準空白。
- A詳: 隨代理普及，工具整合分散且成本高。MCP提供通用模型：工具能力宣告、上下文交換、安全授權(OAuth)、客戶端/伺服端協調。它像USB之於設備：標準化連接，擴大可用工具與生態，讓代理不再孤島。各產業(如Shopify)已推出MCP，推動其成為事實標準。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q11, C-Q8, A-Q14

Q16: 什麼是非同步Agent工作？
- A簡: 將任務以issue/任務單外包給代理在背景處理，人類以PR/審查節點介入。
- A詳: 非同步工作把互動集中在關鍵節點：建立任務(含詳細prompt/規格)，代理在沙箱拉碼、建置、測試、開PR、更新任務，可能耗時15–30分鐘或更久。人類批次審核合併。此模式可水平擴展，多代理並發處理，極大化人類時間效率。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: B-Q10, C-Q6, D-Q4

Q17: IDE配對編程與非同步代理外包有何差異？
- A簡: IDE強互動、同步迭代；非同步以平台協作、批次審核、可水平擴展，更適合長任務。
- A詳: 配對編程像與助手並肩工作，速度快但佔用人類專注。非同步外包以平台(如GitHub/Azure DevOps)承接任務，代理獨立處理後送審，人僅在關鍵決策點介入。隨推理延長與任務增大，非同步成主流，可同時養多代理，提升整體吞吐量。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: B-Q10, C-Q6, D-Q4

Q18: 為何Accessibility成為通用介面？
- A簡: AI以無障礙語意理解UI，像視障輔助工具般「看懂」元素，精準執行自動操作。
- A詳: 無障礙設計(語意角色、可讀名稱、替代文字)讓UI對螢幕閱讀器與自動化更友善。AI代理不具人眼，需語意化結構才能可靠辨識與操作。Playwright等會依A11y標記抽象DOM，提升定位精準度。未來AI大量使用應用程式，A11y亦成為AI的標準入口。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: B-Q12, C-Q7, D-Q1

Q19: Playwright MCP與可及性標記的關係？
- A簡: Playwright會精簡DOM為語意化YAML，依A11y標記決定重要節點，提升代理操作準確度。
- A詳: 直接輸出HTML常超出context，Playwright會抽象出必要元素。若頁面具備正確ARIA角色、名稱與語意結構，抽象結果更準確，AI能穩定尋找按鈕/表單並操作。反之缺標記易導致定位失敗或誤動作。A11y是自動化與代理操作的關鍵品質基礎。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: C-Q7, D-Q1, B-Q12

Q20: 什麼是「抽象化基礎構件」(Abstracted primitives)？
- A簡: 代理運作所需的標準服務：身分驗證、計費、持久化儲存等，以雲端化供應。
- A詳: 如同雲時代的PaaS，代理時代也需標準服務支撐：認證/授權、使用量計費、任務/狀態持久化、審計與治理。這些抽象構件由平台統一提供，讓各種代理快速接入並一致運作，形成可管可控的生態基座。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q11, A-Q21, C-Q8

Q21: 為何須超越.env進行秘密管理？
- A簡: 靜態金鑰風險高；代理需按情境動態授權。OAuth等委派機制更安全可審計。
- A詳: .env將權限常駐在環境中，易外洩、越權且難審計。代理多實體、多場景，需要依任務動態授權，限定範圍與時效。OAuth/短期憑證/金鑰保管(Vault)可最小化權限、可追蹤使用、易輪替，符合代理驅動世界的安全實務。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q16, C-Q8, D-Q5

Q22: 為何CLI/Server-side代理成為主戰場？
- A簡: 易整合管線、便於非同步與擴展，能在沙箱自動建置測試部署，成本線性可伸縮。
- A詳: IDE+AI偏重互動，受人類節奏限制。CLI/Server-side代理可嵌入CI/CD、由webhook觸發、在沙箱執行全流程，並行處理多任務。更適合長推理與背景作業，利於資源調度與成本控管，讓「外包式」開發落地。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q9, B-Q19, C-Q6

Q23: AI-native Git中的gen-commit是什麼？
- A簡: 一種生成式提交：綁定prompt與測試，由代理生成與驗證，記錄上下文與審核要求。
- A詳: gen-commit不是單純提交現成變更，而是以prompt+測試為輸入，觸發代理生成程式並自動驗證。提交元資料包含模型/代理、綁定測試、生成bundle與人審要求，讓變更與意圖強綁定。它連結版控與管線，提升可追溯與品質控管。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q2, C-Q2, D-Q8

Q24: 什麼是「AI可讀性」與「工具可用性」？
- A簡: API/UI需有一致語意與明確規格，讓AI能理解、檢索與正確行動，降低幻覺與誤用。
- A詳: AI可讀性指介面設計與規格對AI友善：一致命名、語意清晰、錯誤語義明確、OpenAPI等機器可讀文檔。工具可用性指MCP/工具能力宣告、輸入輸出定義、安全範圍清楚。兩者讓代理能可靠決策與執行，避免推理偏誤。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q11, C-Q8, D-Q10

Q25: 人與AI在開發流程中的角色如何轉移？
- A簡: 人從「撰寫程式」轉為「定義意圖與驗收」，AI負責生成與自驗，瓶頸移到需求與審核。
- A詳: 隨AI生成效能提升，人類價值在需求分析、規格制定、測試設計與驗收判斷。AI負責大量實作與初步驗證。流程瓶頸也自「寫程式」左移到「定義與審核」，促使版控、文件、測試與管線重新設計以支援此分工。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: A-Q3, B-Q1, B-Q10

Q26: 什麼是Agent-ready dashboard？
- A簡: 同時為人與代理設計的儀表板，提供語意清晰資訊與工具，支援AI行動與人決策。
- A詳: Agent-ready dashboard將資料、操作與語意標註設計成對AI可用的工具集合。AI能檢索日志/錯誤/部署資訊、合成視覺(Mermaid)、觸發操作；人則在同一界面審閱與決策。這種共視設計提升人機協作效率。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q6, C-Q3, D-Q9

Q27: 什麼是「虛擬記憶體式」context管理？
- A簡: 將大規模資訊以文件外置，透過MCP需要時載入/寫回，維持上下文精準與可持續。
- A詳: 模型context有限，將長期內容(規範、筆記、任務)存檔，必要時載入片段，完成後寫回更新。像虛擬記憶體在RAM與磁碟間交換，使AI能以有限context處理大型知識。關鍵在切分、索引與檔案型工具協作。
- 難度: 中級

### Q&A 類別 D: 問題解決類（10題）

Q1: 代理找不到「登入」按鈕怎麼辦？
- A簡: 症狀：操作失敗或誤點。原因：缺A11y標記。解法：加入ARIA語意並測試。預防：用Axe檢測。
- A詳: 症狀：代理定位錯誤、重試多次仍失敗。原因：DOM缺可讀名稱/角色/label，Playwright抽象不準。解決：為按鈕/輸入添加role/name/label，圖片加alt；以可視工具檢測修正；重跑自動化。預防：建立A11y規範與PR檢查，確保語義化HTML。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: C-Q7, B-Q12, A-Q18

Q2: AI輸出與需求不符如何診斷與修正？
- A簡: 症狀：功能偏離。原因：context不足或規格模糊。解法：補文件/測試；TDD分步導引。
- A詳: 症狀：生成結果偏題或漏需求。原因：指令不清、文件缺段、測試不足。解決：完善instructions與需求文件、增補測試案例、用TDD分小步迭代；在gen-commit綁定測試。預防：文件作單一真相、PR門檻要求測試與文件。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: C-Q1, C-Q9, B-Q20

Q3: context window溢出導致資訊遺失怎麼辦？
- A簡: 症狀：AI忘舊事。原因：上下文過長。解法：檔案化長期內容，RAG/MCP分段載入。
- A詳: 症狀：長對話失憶、答非所問。原因：context容量限制。解決：以dev-notes/tasks等檔案存長期內容，建立索引，必要時載入片段；結束後寫回更新。預防：切分文件與標註meta、避免整段貼入。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: C-Q5, B-Q13, A-Q27

Q4: 非同步工作卡住或PR品質差如何改善？
- A簡: 症狀：PR常退件。原因：缺測試/指南。解法：前置TDD、品質門檻；批次審核策略。
- A詳: 症狀：PR頻繁修改、合併慢。原因：需求不清、測試不足、代理自由度太高。解決：在instructions要求測試與風格，建立管線門檻；採TDD引導；審核集中時間段批量處理。預防：平台化任務模板與PR檢查清單。
- 難度: 高級
- 學習階段: 進階
- 關聯概念: C-Q6, C-Q9, B-Q10

Q5: 秘密外洩或越權(.env)如何處置？
- A簡: 症狀：憑證洩露。原因：靜態金鑰。解法：改OAuth與Vault、短期令牌。預防：最小權限。
- A詳: 症狀：環境檔外洩導致服務被濫用。原因：長期金鑰無時效/範圍。解決：收回金鑰、審計使用；改用Vault管理；代理按任務OAuth授權短期令牌；設置scope與TTL。預防：禁止高權限.env、定期輪替與密碼掃描。
- 難度: 高級
- 學習階段: 進階
- 關聯概念: C-Q8, B-Q16, A-Q21

Q6: MCP工具權限錯誤或越權怎麼辦？
- A簡: 症狀：工具執行不當。原因：scope設置不當。解法：重設OAuth範圍與審計。預防：最小授權。
- A詳: 症狀：代理執行超出預期範圍動作。原因：授權過寬或未綁定任務上下文。解決：重新定義scope、添加細粒度權限；啟用審計與告警；在工具描述明確I/O與限制。預防：按任務動態授權與TTL。
- 難度: 高級
- 學習階段: 進階
- 關聯概念: B-Q11, C-Q8, A-Q20

Q7: IDE/CLI代理在沙箱無法建置怎麼辦？
- A簡: 症狀：建置失敗。原因：工具鏈缺失或依賴未備。解法：容器化dev環境。預防：DevContainer。
- A詳: 症狀：代理拉碼後編譯/測試失敗。原因：環境不一致、缺依賴。解決：提供容器化開發環境(DevContainer/Docker)、宣告依賴；在管線提前安裝。預防：環境即程式，版本鎖定與快取策略。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q19, C-Q6, A-Q22

Q8: 文件與程式碼背離如何修復？
- A簡: 症狀：文不對碼。原因：未同步更新。解法：PR門檻與doc-check。預防：AI生成同步文件。
- A詳: 症狀：文件描述與實作不一致。原因：變更未更新文件。解決：設定PR拒絕缺文件；加入doc-lint與連結檢查；代理在生成後同步更新文件片段；回溯元資料。預防：文件作單一真相與審核清單。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q20, C-Q10, A-Q11

Q9: AI UI體驗不佳如何診斷？
- A簡: 症狀：用戶不滿。原因：意圖捕捉差。解法：事件追蹤+AI評分；調整工具與提示。
- A詳: 症狀：錯誤引導、資訊不相關。原因：事件未回報、上下文不足。解決：強化事件模型、記錄關鍵操作；在關鍵節點讓代理輸出滿意度分數與文字；分析數據與文字調整工具、提示與流程。預防：持續迭代與可及性優化。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: C-Q4, B-Q18, A-Q8

Q10: RAG檢索錯誤答非所問如何改善？
- A簡: 症狀：引用不相關。原因：索引/分段差。解法：優化chunk與metadata；評估迭代。
- A詳: 症狀：AI引用錯段落或過時內容。原因：文件結構混亂、分段與標註不良。解決：重構文件、增加標頭/標籤/向量索引；在管線加入檢索評估；必要時以MCP直接取最新資料。預防：文件治理與版本管理。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q14, C-Q5, A-Q13
