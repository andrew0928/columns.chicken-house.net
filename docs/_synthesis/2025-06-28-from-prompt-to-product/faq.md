# 從 Prompt 到 Product

## 問題與答案 (FAQ)

### Q&A 類別 A: 概念理解類

A-Q1: 什麼是「從 Prompt 到 Product」？
- A簡: 以提示驅動，把創意、代碼、流程與評測串起，從點子穩定交付可維運的產品的方法論。
- A詳: 「從 Prompt 到 Product」是一套將生成式 AI 的能力落實到軟體產品的完整方法論。核心是在需求、限制、架構、驗證與運維等環節，用可版本控的提示（prompt）與流程把關，形成可重複、可度量的交付鏈。它強調從 PoC 跨越到可維運的產品化，涵蓋協作模式（如 vibe coding）、度量指標、治理與安全、CI/CD、離線評測與人機協同，讓 AI 產出的價值可被驗證、落地與持續改進。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: A-Q2, A-Q5, B-Q2

A-Q2: 什麼是 Vibe Coding？
- A簡: 以長提示與多輪對話驅動的協作式開發模式，將需求、限制、測試聚合於可版本控的 prompt 檔。
- A詳: Vibe Coding 是一種以 LLM 為共創夥伴的開發模式。開發者在「prompt 檔」中描述目標、範例、架構、邊界與測試，再透過多輪指令與工具使用讓模型產生與修改程式碼。它強調上下文一致性、可回溯性與團隊協作，通常結合 PR 工作流、測試與離線評測，構成「Prompt→Branch→PR→CI→Review→Release」的閉環。適合新功能探索、樣式迭代與跨層整合。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: A-Q3, B-Q1, B-Q2

A-Q3: 什麼是 Prompt 檔（Prompt File）？
- A簡: 以文字定義需求、規範、範例與測試的可版本控制說明檔，驅動模型一致輸出。
- A詳: Prompt 檔是把產品需求、限制、架構指引、風格規範、範例輸入輸出與測試期待等，系統化寫成可追蹤的文本資產。它常置於專案根目錄並進 Git，供 AI 代理或人員重複使用。透過版本控制、審核與測試，能在多人協作下維持上下文一致，降低模型漂移與人為偏差，形成「提示即規格」與「提示即測試」的工程化基礎。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: B-Q3, C-Q2

A-Q4: 為什麼要從 PoC 走向 Product 化？
- A簡: PoC 驗證可行，產品化補齊穩定性、治理、度量與維運，才能持續創造價值。
- A詳: PoC 強調快速驗證想法，而產品化則要求可重複、可維護與可監控。從 PoC 走向 Product，需要導入版本控制、CI/CD、測試與離線評測、風險與合規治理、遙測與回饋迴路，並以量化指標跟蹤交付品質與效益。如此才能將一次性成果轉化為組織可持續擴展的能力，避免停留在示範層級。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: A-Q5, B-Q2, B-Q16

A-Q5: 為什麼 AI 導入需要量化指標？
- A簡: 指標使效益可見、風險可控，利於決策、資源配置與持續改善。
- A詳: 量化指標能將主觀體感轉化為客觀證據，促進決策與溝通。常見包含 DORA 延伸（需求到 PR 時間、PR 接受率、缺陷密度）、生成品質（測試通過率、評測分數）、營運效益（回覆時間、處理成本、CSAT）與風險合規（毒性、洩漏事件）。同時要區分前導指標（如自動化覆蓋率）與結果指標（如故障率），形成可觀測與可優化的閉環。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q20, D-Q10

A-Q6: Prompt Engineering 與軟體架構的差異？
- A簡: 前者聚焦模型行為塑形，後者關注系統分解與非功能需求，兩者需協同。
- A詳: Prompt Engineering 是透過指示、範例與約束塑造模型行為，提高一致性與品質；軟體架構則處理模組劃分、耦合邊界、可維運性與安全性。兩者並非替代，而是互補：優良架構提供清晰邊界與可測單元，讓 prompt 更穩定；高品質 prompt 減少系統複雜度與返工。產品化需同時管理提示資產與架構演進。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q1, B-Q2

A-Q7: PoC 與生產級 AI 系統有何差異？
- A簡: PoC 快速示範；生產系統需可靠、可監控、可治理並支援持續交付。
- A詳: 生產級系統除功能正確，還需達成穩定性、可觀測性、安全與合規、資料治理、成本控管與災難復原等非功能需求。AI 場景另需離線/在線評測、模型與提示版本治理、風險防護（毒性、幻覺）、人工審核與回饋迴路。這些能力讓系統可長期營運與迭代，而非一次性展示。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: B-Q16, B-Q17

A-Q8: Vibe Coding 的核心價值是什麼？
- A簡: 降低溝通成本、提升迭代速度，讓需求與程式同步進化並可審核追蹤。
- A詳: Vibe Coding 把需求、限制、風格與測試融為可版本控的資產，縮短從需求到可執行代碼的距離。透過 PR 與測試串接，團隊可快速對齊期望、回溯決策、度量改動影響。它讓探索與工程化共存：既能高速度嘗試、也能在守護欄下安全上線，提高整體交付效率與品質。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: B-Q1, B-Q2

A-Q9: 初學者在 AI 時代應該學什麼？
- A簡: 穩固計算與網路基礎，學程式設計、系統化思維、資料素養與人機協作。
- A詳: 建議打底計算思維、資料結構、網路與系統基礎；精進一至兩種語言與測試習慣；理解 Git、CI/CD 與可觀測性；學會寫好 prompt、建立小型 RAG、用評測驗證；培養產品與使用者觀點。AI 擅長生成，價值在於能把問題拆解、驗證假設、按約束整合，並負責最終品質與風險。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: A-Q2, B-Q6

A-Q10: 開發者角色會被 AI 取代嗎？
- A簡: 取代的是低價值重複工作；能善用 AI 的工程師將提升影響力與產出。
- A詳: AI 擅長樣板與重複性任務，對初階工作有衝擊；但產品化仍需需求釐清、架構設計、風險治理、資料策展、評測與協作。會被取代的多是拒絕升級工具鏈與思維者。擁抱 AI 的工程師，能以更高層抽象解決問題，將產出擴大到設計、驗證與運維層次，成為槓桿更高的角色。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: A-Q9, B-Q1

A-Q11: 何謂 Monorepo 與多存放庫在 Vibe Coding 上的差異？
- A簡: Monorepo 促進前後端與 prompt 共演；多庫分離清晰但跨庫上下文較難協同。
- A詳: Vibe Coding 需維持上下文一致與跨層整合。Monorepo 讓前後端、infra 與 prompt 檔同庫演進，易於共享契約、測試與評測，適合中小團隊快速協作。多庫有清晰邊界與權限，但需加強跨庫契約、語義版本與自動化同步。選擇取決於團隊規模、治理需求與部署模型。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q8, C-Q3

A-Q12: 前後端分離專案是否該整合以利 Vibe Coding？
- A簡: 若 AI 需跨層生成與整測，整合或 Monorepo 更利於上下文共享與驗證。
- A詳: 當生成會同時觸及 API、UI 與測試時，維持同一上下文能提升一致性與效率。可用 Monorepo 或透過子模組、契約測試與共享 prompt 套件折衷。若分離，應加強 API 合約、Mock 與端到端測試，避免跨層漂移。實務以整測需求與治理邊界為準。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: A-Q11, C-Q3

A-Q13: 何時開新 Prompt 檔，何時修改既有？
- A簡: 新能力或上下文大改用新檔；小修小補或同一能力延伸則維持原檔。
- A詳: 若功能範疇、輸入輸出契約、使用者角色或規範大幅變動，應開新檔保持語義清晰與回溯性；同一能力的 bug 修、文案調整、規則微調可留在原檔並標註版本。可採「一能力一檔」與語義化版本策略，搭配變更日誌與測試，確保影響可控與審核可依。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: B-Q3, C-Q6

A-Q14: Vibe Coding 在維運（SRE/Operations）有何應用？
- A簡: 用 AI 助理做告警彙整、事故初診、Runbook 建議與變更摘要加速回復。
- A詳: 在維運，AI 可整合監控與日誌，將告警關聯、生成事故時間線、提供排查步驟與 Runbook 片段，並在變更單與 PR 中生成風險摘要。搭配權限與審核流程，形成「AI 建議、人類決策」模式，縮短 MTTR，並將處理知識回饋知識庫，打造營運數據飛輪。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q21, C-Q9

A-Q15: 面對高複雜產業知識，Vibe Coding 是否受限？
- A簡: 需以資料策展、RAG/工具化補齊；單靠通用模型會受限且易幻覺。
- A詳: 複雜 domain 需將知識庫結構化，透過檔案切片、向量索引與檢索器優化提供權威上下文；對關鍵規則做工具化或函式調用，避免自由生成；用離線評測約束品質，並設置人工審核。如此，Vibe Coding 仍能帶來速度優勢且可控地擴展到企業級需求。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q4, B-Q5

A-Q16: 導入 AI 的主要風險有哪些？
- A簡: 隱私洩漏、幻覺、偏見合規、成本與供應風險、可維運性不足。
- A詳: 風險包括敏感資料外洩、模型幻覺導致錯誤決策、偏見與法規不符、非功能需求不足導致不穩、供應端變更（API、費率、限額）及成本失控。需以資料最小化、遮罩與審計、離線/線上評測、守護欄、費用配額、可觀測性與備援策略系統化管理。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q10, B-Q19

A-Q17: 什麼是 DORA 指標的 AI 延伸版？
- A簡: 在四指標外，加入 PR 接受率、生成測試通過率、評測分數與風險指標。
- A詳: 典型 DORA 有部署頻率、變更前置時間、變更失敗率、恢復時間。AI 延伸加入：Prompt→PR 前置時間、PR 接受率、生成代碼測試通過率、評測分數（事實性、害群性）、觀測到的毒性/洩漏事件、推理成本與延遲。以此量化 AI 對交付與風險的影響。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: A-Q5, B-Q20

A-Q18: 什麼是 AI Evals？為何需要？
- A簡: 以標準任務與資料集評測模型/提示品質，建立可比較的品質基線。
- A詳: AI Evals 是針對特定任務（問答、生成、分類）設計的離線評測，搭配標準資料與客觀/半主觀評分，建立可重複的品質度量。它補足線上指標的不足，讓提示或版本變更可在安全環境先驗證，減少回歸風險。常與 A/B 測試與人審結合，形成品質守門機制。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q6, C-Q7

A-Q19: RAG 與微調（Fine-tuning）的差異？
- A簡: RAG注重外掛知識檢索，微調著重內化風格/任務；常搭配使用。
- A詳: RAG 透過檢索外部權威資料作為上下文，降低幻覺並保持資料更新；微調則改變模型權重，提升特定任務與風格一致性。若需求常變、資料敏感，RAG 更彈性；若需要嚴格結構輸出或術語風格一致，微調有利。實務常以 RAG 為主、少量微調輔助。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: B-Q4, B-Q13

A-Q20: 什麼是 PromptOps（Prompt Operations）？
- A簡: 以工程化方法治理提示：版本、審核、測試、監控與回饋閉環。
- A詳: PromptOps 將提示視為一級資產，透過 Git 版本控、PR 審核、離線評測、灰度與 A/B、線上遙測與回饋收集、定期重訓或調參，形成從開發到運營的治理流程。目標是提高可預測性、降低回歸並加速迭代。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q3, B-Q16

A-Q21: 什麼是守護欄（Guardrails）與政策治理？
- A簡: 在輸入/輸出/工具層設限與審核，避免違規、洩漏與危害。
- A詳: 守護欄包含敏感詞與 PII 遮罩、提示注入防護、輸出檢測（毒性、仇恨）、工具調用白名單與參數約束、人工審核閘道與審計記錄。政策治理定義可接受用途、資料處理規則與合規要求，並以技術與流程共同落實，降低風險。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q10, D-Q8

A-Q22: 什麼是人類在回路（HITL）？
- A簡: 關鍵節點引入人工審核與決策，平衡風險與效率並累積知識。
- A詳: HITL 在高風險或高不確定性場景，引入人審作為最後把關，例如合約生成、醫療建議、上線前評測、事故處置建議。審核結果回饋模型與提示，形成學習閉環。藉此在保留 AI 效率的同時，維持品質與合規。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: B-Q11, A-Q21

A-Q23: Copilot 即時補全與 Agent/工作流的差異？
- A簡: 補全偏局部輔助；Agent/工作流可跨步驟、跨工具完成任務。
- A詳: 即時補全強於本地上下文的速度與便捷，適合微任務；Agent/工作流可規劃、檢索、工具調用與多輪校正，適合復雜任務與跨系統整合。產品化常把兩者結合：日常用補全提效，關鍵流程交由受控 Agent 工作流。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: B-Q14, B-Q15

A-Q24: 什麼是資料飛輪（Data Flywheel）？
- A簡: 以真實使用與審核回饋持續改進資料、提示與模型，形成正循環。
- A詳: 透過線上遙測與 HITL 收集實際輸入輸出、錯誤與修正，整理為高品質資料與測試案例，反向優化檢索器、提示與模型參數。持續迭代使品質越來越好，成本下降、風險降低。關鍵在可觀測性、標註流程與自動化管道。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q6, B-Q7

### Q&A 類別 B: 技術原理類

B-Q1: Vibe Coding 的運作原理是什麼？
- A簡: 以提示檔定義規格，透過多輪對話與工具鏈生成、修正並驗證代碼。
- A詳: 原理是將需求、限制、範例與測試收斂為可機器讀的提示檔，供 LLM 以多輪互動生成候選方案；再以工具（測試、靜態分析、型別檢查、格式化）自動校正，必要時回饋錯誤給模型二次修復。流程包含上下文管理、工具調用、回饋迴路與人工評審，確保輸出可合併。核心組件：Prompt 檔、上下文蒐集器、工具執行器、CI 測試與 PR 審核。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: A-Q2, C-Q5

B-Q2: 「Prompt→PR」產品化流水線如何設計？
- A簡: 提示出分支代碼，跑測試與評測，通過審核後合併並部署。
- A詳: 流水線步驟：1) 從 Prompt 檔與上下文拉取目標；2) 生成/修改代碼至功能分支；3) 觸發 CI 跑單元/整測與離線評測；4) 自動產出 PR 說明與風險摘要；5) 人工審核與守護欄檢查；6) 合併與部署，並回寫成效至遙測。核心組件：Prompt 版本庫、生成代理、CI/CD、Evals 平台、可觀測性。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: A-Q4, A-Q20

B-Q3: Prompt 版本控與治理怎麼做？
- A簡: 用 Git 管理語義版本、PR 審核、變更日誌與自動評測把關。
- A詳: 將 Prompt 視為一級資產：採語義化版本（MAJOR.MINOR.PATCH）、PR 模板強制描述影響範圍、Changelog 記錄、對每次變更跑回歸評測與敏感詞檢查。建立標籤對應線上配置，支援灰度與回滾。核心組件：Git、PR 工作流、CI Evals、配置管理。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: A-Q3, C-Q4

B-Q4: 典型 RAG 架構如何運作？
- A簡: 切片與嵌入索引，檢索相關片段與查詢重寫，擴充上下文再生成。
- A詳: 流程：1) 資料切片與清洗；2) 產生向量嵌入建索引；3) 查詢前處理（意圖/擴寫/改寫）；4) 檢索器（相似度/重排）取回片段；5) 構建提示上下文；6) 生成與後處理（引用、結構化）；7) 遙測與回饋。核心組件：向量庫、檢索器、重排器、提示模板與觀測。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: A-Q19, C-Q8

B-Q5: 如何注入複雜領域知識？
- A簡: 結構化知識、權威來源優先，搭配 RAG、工具化與校驗規則。
- A詳: 將規範/條款/流程整理為文件與結構化表，建立權威資料管道；以 RAG 將最新內容帶入上下文，對關鍵規則做工具化（函式調用/查詢接口），輸出使用格式與約束校驗（schema/regex）。離線評測與人審作最終把關。核心：知識庫、RAG、工具與驗證器。
- 難度: 高級
- 學習階段: 進階
- 關聯概念: A-Q15, B-Q14

B-Q6: 離線評測與 A/B 測試的機制？
- A簡: 先離線回歸守門，再線上灰度對照，觀測指標差異決策上線。
- A詳: 變更先跑離線回歸（基準題庫、指標如事實性/嚴謹度/毒性），過門檻才可進入線上灰度。A/B 測試分流流量，跟蹤 CSAT、成功率、延遲與成本，達顯著差異後合併。核心：資料集、評分器、流量控制與遙測管道。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: A-Q18, C-Q7

B-Q7: 互動遙測與可觀測性如何設計？
- A簡: 全鏈路記錄輸入輸出、檢索內容、評分與決策，支援追溯與優化。
- A詳: 設計事件模型，記錄使用者輸入、檢索片段、模型配置、輸出、評測分與人審結果；關聯上游版本（prompt/模型/資料）與下游效果（點擊、成功率）。提供查詢面板、告警與抽樣回放，支援資料飛輪。核心：事件匯流排、日誌/指標/追蹤、資料倉與面板。
- 難度: 高級
- 學習階段: 進階
- 關聯概念: A-Q24, B-Q20

B-Q8: 面向 Vibe Coding 的 Monorepo 架構？
- A簡: 以包/工作區劃分前端、後端、prompt、測試與 infra，統一 CI。
- A詳: 採工作區管理（如 pnpm/turbo/lerna 或 Bazel），將前端、後端、共用 SDK、prompt 套件、測試資料與基建 IaC 分倉夾層化；統一 lint/test/build 與 Evals。暴露清晰契約（API/型別/Schema）與跨包版本關係，利於跨層生成與整測。核心：包管理、CI pipeline、契約與測試框架。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: A-Q11, C-Q3

B-Q9: 生成代碼的測試策略如何設計？
- A簡: 以契約與範例驅動，結合單元、合約、端到端與回歸評測。
- A詳: 將輸入輸出契約與範例內嵌於 prompt 與測試；用單元測試保障函式級正確，契約測試鎖定 API/Schema，端到端覆蓋跨層行為；對文字生成另設離線評測與金標比較。失敗案例反哺測試與 prompt。核心：測試金標、契約工具、E2E 與 Evals。
- 難度: 高級
- 學習階段: 進階
- 關聯概念: C-Q7, D-Q4

B-Q10: 安全與合規守護欄的技術機制？
- A簡: 輸入遮罩、輸出檢測、工具白名單、審計與隔離執行。
- A詳: 輸入層做 PII 遮罩與提示注入防護；輸出層檢測毒性/仇恨與資料洩漏；工具層限定可調用 API 與參數；系統層做權限分割、網路隔離與審計。搭配人審閘道與事件告警。核心：敏感資訊偵測、策略引擎、審計流水。
- 難度: 高級
- 學習階段: 進階
- 關聯概念: A-Q21, D-Q8

B-Q11: 人類在回路（HITL）工作流如何落地？
- A簡: 對風險步驟加審核節點、模板化決策依據並沉澱知識。
- A詳: 定義高風險步驟（如法務敘述、部署變更），於流程中插入審核任務；AI 生成候選與風險摘要，人員按清單審核並選擇/編輯。結果保存為知識與測試樣本，回饋模型或提示。核心：工作流引擎、審核 UI、審計與資料飛輪。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: A-Q22, C-Q9

B-Q12: Prompt-as-Code 的實踐要點？
- A簡: 用程式與模板管理提示，抽象變數、參數化與測試化。
- A詳: 將提示拆分模板與變數，透過程式組裝上下文；建立型別與 Schema 約束，加入測試與快照比對；以套件形式重用，支援語義版本與依賴管理。核心：模板引擎、型別/Schema、測試框架與封裝。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: C-Q2, C-Q4

B-Q13: 何時選 RAG、何時微調（決策準則）？
- A簡: 資料常更新選 RAG；格式/風格穩定或樣本充足時考慮微調。
- A詳: 若知識變動頻繁、需權威引用與控風險，RAG 優先；若輸出格式嚴格、風格一致或樣本量大且資料可用，微調加分。亦可先 RAG 快速落地，後期以輕量微調優化關鍵任務。核心：資料動態性、合規、成本與維護性。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: A-Q19, C-Q8

B-Q14: 工具調用（Function Calling）如何工作？
- A簡: 模型先決定意圖，再以結構化參數呼叫外部函式取得權威結果。
- A詳: 提供函式簽名與描述，模型根據意圖產出參數，系統以結構化呼叫執行器並回填結果至對話，實現「檢索或計算→再生成」。此法能降低幻覺、確保數據準確。核心：函式描述、調用代理、執行沙箱與回填策略。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q5, C-Q8

B-Q15: 代理/圖工作流（Agents/Graphs）設計原理？
- A簡: 將任務分解為節點與邊，節點可檢索、推理或調用工具並循環自我修正。
- A詳: 以任務圖定義狀態、行為與轉移，節點可做規劃、檢索、執行與評估；透過迭代與停止條件控制成本與品質。適合長鏈推理與跨系統任務。核心：狀態機、規劃器、記憶與觀測。
- 難度: 高級
- 學習階段: 進階
- 關聯概念: A-Q23, B-Q16

B-Q16: AI 元件的 CI/CD 有何不同？

- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q2, B-Q6

B-Q17: 非決定性輸出的測試如何穩定？
- A簡: 用容忍度、正則/Schema 檢驗與快照更新流程管理。

B-Q18: 如何處理 Token 限制與上下文管理？

### Q&A 類別 C: 實作應用類（10題）

C-Q1: 如何從零啟動一個「Prompt 到 Product」專案？
- A簡: 明確目標與指標，建 Prompt/評測/CI 基礎，先小範圍灰度迭代。
- A詳: 步驟：1) 定義業務目標與成功指標；2) 建 Monorepo 或清楚契約；3) 建立 prompt 檔與模板；4) 設 CI（測試+Evals）；5) 首批金標資料；6) 小流量灰度與儀表板；7) HITL 與回饋機制。注意只做最小可行閉環，週週迭代。最佳實踐：版本治理、可觀測性先行。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: B-Q2, C-Q4

C-Q2: 如何撰寫與結構化 Prompt 檔？
- A簡: 採模板化區塊：目標、規則、上下文、範例、輸出格式與測試鉤子。
- A詳: 建議結構：header（角色/語氣）、任務目標、邊界/禁忌、輸入輸出定義、Few-shot 範例、評分準則、錯誤處理與工具說明。以變數注入上下文，輸出以 JSON Schema 約束。代碼片段：
// prompt.md.ptmpl
# Role: {{role}}
## Goal
{{goal}}
## Constraints
- {{constraints}}
## Examples
{{#each examples}}...{{/each}}
## Output Schema
{{jsonSchema}}
注意同步測試樣本、維護 Changelog。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q12, C-Q7

C-Q3: 如何整合前後端與 Prompt 在一個 Monorepo？
- A簡: 以工作區管理前端/後端/prompt 套件，統一測試與 Evals。
- A詳: 步驟：1) 建 repo 與工作區；2) packages/web、api、prompts、shared；3) 設統一 lint/test；4) E2E 與 Evals pipeline。範例設定（pnpm-workspace.yaml）:
packages:
  - packages/*
  - apps/*
並在 prompts/ 作為 NPM 套件被 web/api 共用。注意清楚版本與 API 契約，避免跨包隱性耦合。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q8, A-Q12

C-Q4: 怎麼用 Git 與 CI 建立 Prompt 版本治理？
- A簡: 每次改動走 PR，CI 自動跑 Evals 與敏感檢查，審核後才可合併。
- A詳: 步驟：1) prompt/ 目錄；2) PR 模板要求影響說明；3) CI 任務跑 evals:run、toxicity:scan；4) 未過門檻阻擋合併；5) 釋出標籤對應線上配置。GitHub Actions 例：
- name: Evals
  run: npm run evals -- --threshold 0.8
- name: Safety
  run: npm run safety:scan
最佳實踐：語義版本、Changelog、自動回滾。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q3, B-Q16

C-Q5: 如何用 Vibe Coding 開發新功能？
- A簡: 新建 prompt 檔，生成分支實作，回合修復通過測試後提 PR。
- A詳: 步驟：1) 定義需求與測試；2) 建 feature prompt；3) 用代理生成代碼至 feature branch；4) 本地跑測試與修復輪；5) 開 PR，CI/Evals；6) 審核合併。注意將失敗案例加入 prompt 與測試，保持上下文一致。最佳實踐：小步快跑、單一責任 Prompt。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: B-Q1, B-Q9

C-Q6: 既有功能小修改要不要新開 Prompt 檔？
- A簡: 小改動維持原檔並標註版本；大範圍行為變動才新開。
- A詳: 判斷準則：契約未變且屬文案/規則微調→沿用；若輸出結構或流程變更→新開。步驟：更新原檔、補測試、跑回歸 Evals、PR 說明影響。最佳實踐：保持「一能力一檔」與語義版本。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: A-Q13, B-Q3

C-Q7: 如何建立離線評測與門檻？
- A簡: 準備金標資料，設評分器與最低門檻，CI 自動阻擋回歸。
- A詳: 步驟：1) 收集實際問答/輸出作金標；2) 指標設計（事實性、結構正確率、毒性）；3) 建評分器（規則/模型評分/人工稽核）；4) CI 跑 evals；5) 儀表板追蹤。程式片段（假想）:
npm run evals -- --dataset=qa.json --minScore=0.8
注意樣本持續擴充與抽樣覆蓋。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q6, B-Q17

C-Q8: 如何接 RAG 與向量資料庫？
- A簡: 導入切片與嵌入，查詢時檢索重排，將片段注入提示上下文。
- A詳: 步驟：1) 以文本分段器切片；2) 以選定模型產嵌入寫入向量庫；3) 查詢改寫與檢索；4) 用重排器提升相關性；5) 注入提示並引用來源。程式片段（pseudo）:
const ctx = retriever.search(q).rerank().topK(5)
prompt = render(ctx, q)
注意嵌入與生成模型搭配、更新與回歸評測。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q4, B-Q14

C-Q9: 如何將 Vibe Coding 應用於維運流程？
- A簡: 建立告警匯整 Prompt 與 Runbook 檢索，導入審核閘道與追蹤。
- A詳: 步驟：1) 蒐集告警與日誌；2) 建 RAG 索引 Runbook；3) 設告警摘要/處置建議 Prompt；4) 工單系統串連 HITL；5) 儀表板與質量抽檢。注意權限最小化與審計，避免自動化直接變更。最佳實踐：先建議、後半自動。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q21, A-Q14

C-Q10: 如何定義 AI 導入量化指標並建儀表板？
- A簡: 選前導/結果指標，布署遙測，建可視化面板與告警門檻。
- A詳: 步驟：1) 對齊業務目標；2) 決定指標（DORA 延伸、Evals、成本、CSAT）；3) 事件模型設計；4) 收集至倉庫；5) 建面板（如 Grafana/Looker）；6) 設告警與週會檢視。程式片段：定義事件 schema 與 ETL 任務。注意維持可比性與統計顯著性。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q7, B-Q20

### Q&A 類別 D: 問題解決類（10題）

D-Q1: 生成結果每次不一致怎麼辦？
- A簡: 固定隨機種/溫度、約束輸出結構、用測試與快照治理非決定性。
- A詳: 症狀：同一輸入輸出波動。原因：溫度過高、上下文漂移、無結構約束。解法：降低溫度/採 deterministic 模式、加入 JSON Schema 或正則、關鍵段落改工具化、建立容忍區間與快照審核。預防：固定模型與版本、上下文模板化、回歸評測把關。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q17, B-Q18

D-Q2: 模型對領域事實產生幻覺如何處理？
- A簡: 導入 RAG 與權威資料、工具查證與引用來源，人審高風險輸出。
- A詳: 症狀：自信但錯誤的專業敘述。原因：缺少權威上下文、模型偏差。解法：建 RAG 索引、關鍵資料工具化查詢、要求引用與鏈接、對高風險場景啟 HITL。預防：資料策展、評測題庫涵蓋棘手案例、模型與提示定期回歸。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q4, B-Q5

D-Q3: Prompt 過長觸發 Token 限制怎麼辦？
- A簡: 上下文檢索化、摘要與動態裁剪，減少冗語並結構化輸出。
- A詳: 症狀：長提示報錯或成本暴增。原因：冗長描述與非結構上下文。解法：切換為 RAG，只帶相關片段；用分層摘要；模板化與佈局壓縮；工具化轉移計算。預防：建立上下文編排器、監控 token 使用。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q18, C-Q8

D-Q4: CI 測試常因生成代碼失敗？
- A簡: 明確契約與測試、降低溫度、加入回合修復與本地快照校驗。
- A詳: 症狀：PR 經常紅燈。原因：缺乏契約/範例、非決定性、邏輯漏洞。解法：在 prompt 內嵌契約/範例、使用 Schema 驗證、讓代理先本地跑測試與修復、再提 PR。預防：更小粒度的任務拆分與金標覆蓋。
- 難度: 高級
- 學習階段: 進階
- 關聯概念: B-Q9, B-Q17

D-Q5: PR 接受率低怎麼提升？
- A簡: 改善需求對齊、增加範例與測試、讓 AI 產出更可審核與可追溯。
- A詳: 症狀：PR 常被退回。原因：需求理解不一、風格不符、測試不足。解法：強化 prompt 之 Do/Don’t、加入範例與架構準則、自動生成 PR 說明與影響分析、跑自動化檢查。預防：建立程式碼規約與模板、持續蒐集退件原因回饋。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q2, C-Q5

D-Q6: RAG 檢索常拿到無關內容？
- A簡: 改良切片與嵌入、重排與查詢改寫，並加權威度與新鮮度。
- A詳: 症狀：回答偏題或過時。原因：切片不佳、嵌入/檢索器不匹配、無重排。解法：語義切片、選擇更適嵌入、加入 BM25+重排、查詢改寫、片段打分與去重。預防：索引版本與回歸評測、自動更新。
- 難度: 高級
- 學習階段: 進階
- 關聯概念: B-Q4, B-Q22

D-Q7: 延遲與成本過高如何優化？
- A簡: 模型分層、上下文裁剪、批次化與快取、關鍵步驟本地化。
- A詳: 症狀：回應慢、費用高。原因：大模型濫用、上下文冗長、重算。解法：前置用小模型、只在關鍵步用大模型；RAG 限 K 值；結果快取；批次嵌入與流水線；工具化計算。預防：成本面板與配額、架構紅線。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q19, B-Q18

D-Q8: 安全與隱私風險如何控管？
- A簡: PII 遮罩、策略掃描、工具白名單、人審與審計流水，最小權限。
- A詳: 症狀：洩漏或違規輸出。原因：缺乏遮罩、提示注入、權限過大。解法：輸入遮罩與正則、輸出毒性/洩漏掃描、限制工具與網路、人工審核閘道、審計與告警。預防：安全測試與演練、供應商評估、最小化資料。
- 難度: 高級
- 學習階段: 進階
- 關聯概念: B-Q10, A-Q21

D-Q9: Monorepo 合併衝突頻繁怎麼解？
- A簡: 明確分層與邊界、細粒度套件、強化契約與自動化重整。
- A詳: 症狀：多人改動互相踩踏。原因：邊界不清、耦合高。解法：拆分 package、界定 API/Schema 契約、採代碼產生器統一輸出、格式化與 import 排序自動化。預防：代碼所有權與評審路由、CI 阻擋跨層未經審核改動。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q8, C-Q3

D-Q10: 利害關係人質疑 AI 效益，如何說服？
- A簡: 以對齊目標的量化指標與對照實驗，展示風險可控與可回滾。
- A詳: 症狀：主觀抗拒。原因：缺乏數據與風險顧慮。解法：定義前導/結果指標，設 A/B 對照；展示 MTTR/成本/滿意度改善與案例；說明守護欄、回滾與審計；邀請試點共創。預防：持續透明儀表板與週會檢視，建立信任。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: A-Q5, B-Q20
