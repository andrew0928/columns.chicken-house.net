# 從 Prompt 到 Product

## 問題與答案 (FAQ)

### Q&A 類別 A: 概念理解類

Q1: 什麼是「從 Prompt 到 Product」？
- A簡: 以提示為規劃起點，串接設計、生成、測試與部署，將原型轉化為可維運的產品。
- A詳: 「從 Prompt 到 Product」是一種以提示為需求與設計介面，讓生成式 AI參與整個開發生命週期的方法。它包含將想法轉為可測的規格、用 Vibe coding 驅動原型、以提示檔案化管理、加入自動化測試與評估、前後端整合、部署到可觀測與可管控的生產環境。這種方法的重點在於把聊天成果工程化：版本化、可重現、可度量，並用指標與治理把原型提升為可靠、可維運的產品。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: A-Q3, B-Q1, C-Q1

Q2: Prompt 與 Specification 的差異？
- A簡: Prompt面向模型溝通；Specification面向人與系統，強調可測、可重現、邊界與約束。
- A詳: Prompt是與模型互動的語言，偏向敘述目標、語氣與上下文，易於探索但易漂移。Specification（規格）則是面向人與系統的正式描述，包含輸入輸出格式、約束條件、錯誤處理與驗收標準，可被測試與版本管理。實務上，我們常以 prompt 驅動規格草稿，再提煉成可測的 spec，並以提示檔對齊規格與測試，讓生成過程符合工程品質。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: A-Q1, B-Q7, C-Q1

Q3: 什麼是 Vibe Coding？
- A簡: 以高階意圖描述需求，透過 AI輔助生成骨架與細節，人類負責引導與審核。
- A詳: Vibe coding 是以「意圖與感覺」描述目標，再用模型生成初稿與改寫。工程師提供範例、約束與驗收，逐步逼近可運行的程式。它適合探索新功能、建立樣板、重構樣式，效率高但易產生幻覺與不一致，需搭配測試、規格與版本化管理。對大型系統，vibe coding 是加速器而非替代；關鍵在將它嵌入既有工程流程。
- 難度: 初級
- 學習階段: 核心
- 關聯概念: A-Q1, B-Q2, C-Q2

Q4: 為什麼需要將 Prompt 檔案化管理？
- A簡: 讓提示可版本化、可重現、可審計，支援協作、測試與持續改進。
- A詳: 將 prompt 檔案化（Prompt as Code）能把臨時對話轉為受控資產：可追蹤改動、綁定規格與測試、建立審核流程與合規檢查。提示檔可含系統提示、範例、語域、邊界、錯誤案例與評估指標，與程式並存於版本庫。對既有功能的小改動，也應更新相應提示檔與測試，確保生成行為穩定，避免知識散落於聊天記錄。
- 難度: 初級
- 學習階段: 核心
- 關聯概念: C-Q1, C-Q3, B-Q3

Q5: 生成式 AI 在軟體開發中的核心價值？
- A簡: 提升探索速度與覆蓋，自動化樣板與重複工作，加速迭代與知識萃取。
- A詳: 生成式 AI最強在於加速探索與擴散：快速產生多版本方案、生成骨架與樣板、撰寫測試、整理文件與變更摘要。它能幫助跨職能協作，讓需求到原型縮短時間，並用度量指標驗證效益。局限在穩定性與可控性，故需搭配規格、測試、RAG/工具接合與治理，將加速能力轉化為可靠產能。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: A-Q1, B-Q7, C-Q2

Q6: 前後端分離與整合在 Vibe Coding 的取捨？
- A簡: 分離利於邊界與獨立部署；整合利於提示協同、端到端測試與快速迭代。
- A詳: 前後端分離可保持清楚契約、獨立版本與可擴展性；但在 vibe coding 情境下，端到端迭代與整合測試很重要。建議採 monorepo 管理提示與契約，工具層維持分離，在整合環節提供端到端示例與合約測試。生成流程中，前端/後端共享範例與驗收條件，確保 AI 產物能跨層協同，而非破壞分層設計。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q4, C-Q4, D-Q2

Q7: 企業導入 AI 為何要量化效益？
- A簡: 量化指標使決策可驗證，可持續投資與治理，避免單靠主觀印象。
- A詳: 企業導入 AI 涉及成本、風險與轉型。以量化指標（如交付速度、缺陷率、回歸率、DORA、單次請求成本、覆蓋率、工時節省）能客觀評估試點成效，支持擴張或收斂。指標與目標應綁定業務成果，如縮短上線時間、提升客訴處理效率。量化讓利害關係人共識化，並形成持續改進循環。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: C-Q6, B-Q8, D-Q8

Q8: 什麼是 AI PM？
- A簡: 結合產品管理與生成式AI運用，以資料與指標驅動從探索到交付。
- A詳: AI PM 是能掌握產品策略與生成式 AI 能力的產品經理。其工作包含定義問題與價值、設計資料與知識來源（如 RAG）、制定評估與治理、協調工程與法務安全、設計實驗與 A/B、將 prompt 與規格納入版本與測試流程。AI PM 以指標與學習循環推動「提示到產品」落地。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: A-Q7, B-Q5, B-Q7

Q9: RAG 與微調的差異？
- A簡: RAG以檢索補充上下文；微調改變模型權重，使風格與知識內化。
- A詳: RAG（檢索增強生成）在推理前把相關文件嵌入檢索，餵入模型上下文，優點是即時、易更新、成本低；缺點是受限於上下文長度。微調則以資料訓練模型，使特定風格或領域知識內化，推理更穩定；缺點是成本高、需資料治理與持續維護。企業常以 RAG 為主、微調補強特定場景。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q5, C-Q5, D-Q4

Q10: Agent 與 Pipeline 架構差異？
- A簡: Agent具狀態與工具調用自迭代；Pipeline偏固定流程與明確步驟。
- A詳: Agent 架構讓模型在目標導向下，能保持狀態、規劃子任務並調用工具，自行迭代到達結果；適合探索性或任務不確定。Pipeline 是明確步驟的定序處理，如提取、摘要、評估與回覆，利於可控與測試。實務上常以 pipeline 包裹 agent，限制邊界並設計安全與度量。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q12, B-Q6, D-Q5

Q11: 原型聊天機器人與正式產品差異？
- A簡: 原型驗證想法；產品需可靠、可維運、可測試與合規與成本控制。
- A詳: 聊天原型重在展示能力與收集回饋；正式產品則必須具備穩定 SLA、可觀測、可回滾、可測試與合規安全。從原型到產品需加入提示檔案化、評估與測試、前後端契約、資料治理、成本與延遲管理、使用者體驗與支援流程，並以指標持續改善。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: A-Q1, B-Q8, D-Q5

Q12: 大型企業系統中的 domain knowledge 為何重要？
- A簡: 領域知識定義邊界與術語，影響準確性、合規與流程可控性。
- A詳: 企業級系統牽涉專有術語、法規、流程與歷史資料。沒有充足 domain knowledge，模型容易誤解語義或產生違規建議。解法包含建立知識庫、RAG、術語表、流程卡與範例、將決策點明確規格化。Vibe coding 在此作為加速器，需以資料與規格將領域知識框定到可測範圍。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: A-Q9, C-Q5, D-Q4

Q13: 什麼是 LLMOps？
- A簡: 管理模型與提示的實務：部署、監控、評估、成本與安全治理。
- A詳: LLMOps 是把生成式 AI 納入工程與營運的集合實務。涵蓋提示與規格版本化、資料與嵌入管理、評估與測試、實驗追蹤、部署與回滾、觀測與告警、成本與延遲控制、權限與合規。目標是讓模型與提示像服務一樣可被治理，支撐從原型到產品的可靠交付。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q8, C-Q10, D-Q5

Q14: Prompt Engineering 與 Prompt Orchestration 差異？
- A簡: Engineering著重單一提示品質；Orchestration編排多步流程與資源。
- A詳: Prompt Engineering 是設計與優化單一提示的技巧，包括角色、語域、範例與評估。Prompt Orchestration 則關注跨步驟的提示與上下文管理：檢索、工具調用、狀態與記憶、錯誤恢復、度量與安全。從產品角度，重心由工程走向編排，確保端到端的可控與可測。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: B-Q1, B-Q12, C-Q4

Q15: 為什麼初學者仍需學習程式與工程基礎？
- A簡: 基礎確保理解邏輯與品質，能正確引導AI並驗證與修正其輸出。
- A詳: AI 能加速生成，但無法取代對邏輯、資料結構、測試與架構的理解。初學者若缺基礎，難以辨識幻覺、設計邊界、寫出可測規格與可維運程式。學習重點包括版本控制、測試、API與資料建模、前後端契約、觀測與安全。這些讓人類能成為有效的 AI 導演。
- 難度: 初級
- 學習階段: 基礎
- 關聯概念: D-Q10, C-Q2, B-Q7

Q16: 生成式 AI 導入的風險與治理要點？
- A簡: 幻覺、資料洩漏、合規與偏見需以邊界、審核、度量與隔離控制。
- A詳: 導入風險包含錯誤建議、敏感資料外洩、違反法規、模型偏見與不可重現。治理策略包括權限分層、PII 屏蔽、輸出審核、紅藍測試、離線評估、成本與延遲閾值、變更審核與回滾、記錄與審計。以政策、工具與流程共同確保安全與品質。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: D-Q6, B-Q9, A-Q13

Q17: 從架構師角度看「提示到產品」的關鍵角色？
- A簡: 協調策略、資料、流程與品質，把提示轉為可測、可維運的系統。
- A詳: 架構師需定義目標與邊界、選型模型與工具、設計資料與檢索、建構編排與狀態、制定測試與度量、導入觀測與治理、設計成本與彈性策略。並與 PM、工程、法務、安全合作，把提示實驗變成可靠產品，建立組織的 LLMOps 能力。
- 難度: 高級
- 學習階段: 進階
- 關聯概念: B-Q1, B-Q8, A-Q13

Q18: 指標與 OKR 在 AI 導入的角色？
- A簡: 以清晰目標與量化關鍵成果連結技術行動與業務價值。
- A詳: OKR 將願景轉為具體目標與可量測的結果，如「縮短功能交付週期 30%」或「提高回答準確率至 90%」。指標設計要涵蓋品質、速度、成本與風險，並可在儀表板可視化。OKR 驅動跨部門對齊，確保 AI 導入不是炫技，而是與業務成果連動。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: A-Q7, C-Q6, D-Q8


### Q&A 類別 B: 技術原理類

Q1: 「從 Prompt 到 Product」的端到端流程如何運作？
- A簡: 從意圖到規格、生成、測試、部署、監控，以迭代與度量閉環。
- A詳: 端到端流程包含：1) 定義問題與價值指標；2) 以提示探索並提煉成可測規格；3) Vibe coding 生成骨架與範例；4) 建立提示檔案化與版本控制；5) 編排檢索、工具調用與管線；6) 撰寫自動化測試與離線評估；7) 前後端整合與合約測試；8) 部署、觀測、告警與回滾；9) 依指標迭代優化。核心組件：模型、編排層、資料/RAG、測試評估、監控與治理。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: A-Q1, A-Q14, C-Q10

Q2: Vibe Coding 的技術機制？
- A簡: 以範例與約束驅動生成，透過反饋循環修正與收斂到可運行。
- A詳: 機制包含：1) 系統提示設定目標與風格；2) 提供高品質範例與邊界；3) 生成程式骨架；4) 自動執行測試，回饋錯誤；5) 迭代重寫特定片段；6) 人類審核與重構。關鍵步驟是建立可重現回饋環，並將驗收條件內嵌在提示與測試中。組件：模型、測試框架、版本庫、評估器與執行環。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: A-Q3, C-Q2, B-Q7

Q3: Prompt 檔案化與版本控制的設計？
- A簡: 以結構化格式存放提示、範例、測試連結與指標設定。
- A詳: 設計做法：1) 使用 YAML/JSON/TOML 定義系統提示、變數、Few-shot 範例；2) 將驗收規格與測試案例連結；3) 記錄評估分數與成本；4) 使用 Git 管理分支與審核；5) 建立變更日誌與回滾機制。流程：開分支—修改提示—跑評估—審核—合併。組件：版本庫、CI、評估服務、儀表板。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: A-Q4, C-Q1, C-Q3

Q4: 前後端一體化開發與 monorepo 架構原理？
- A簡: 共享契約與測試，分層封裝實做，端到端管線確保一致性。
- A詳: Monorepo 將前後端、提示與測試共存於同一版本庫。核心：1) 契約（API schema）作為共享單一真相；2) 端到端測試與合約測試；3) 分層目錄與包管理；4) CI 以影響範圍驅動工作流；5) 提示與生成腳本共版。流程：修改—局部建置—測試—部署。組件：包管理器、CI/CD、合約測試框架。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: A-Q6, C-Q4, C-Q10

Q5: RAG 系統的工作機制？
- A簡: 將文件嵌入向量，檢索相關片段，與提示合併供模型生成。
- A詳: 步驟：1) 資料清洗與切片；2) 嵌入向量化並存入索引（FAISS/pgvector）；3) 查詢轉向量，檢索最相似片段；4) 構造提示上下文，加入引用；5) 模型生成；6) 評估與回饋。組件：向量資料庫、嵌入模型、檢索器、重排序器、合成器。關鍵是切片策略、Metadata 與權限控制。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: A-Q9, C-Q5, D-Q4

Q6: 模型工具調用（function calling）的流程？
- A簡: 模型判斷需求，選擇工具，序列化參數，執行並將結果融入回覆。
- A詳: 流程：1) 系統提示宣告可用工具與契約；2) 模型根據意圖產生工具調用；3) 對參數進行結構驗證；4) 執行外部 API/函式；5) 收集結果並餵回模型；6) 輸出最終回覆或下一步計畫。組件：工具目錄、調用器、驗證器、執行沙箱、審計記錄。關鍵：安全邊界與錯誤處理。
- 難度: 中級
- 學習階段: 核心
- 關聯概念: A-Q10, C-Q4, D-Q2

Q7: LLM 評估與自動化測試機制？
- A簡: 離線基準與線上監測並行，使用標註集與規則度量品質。
