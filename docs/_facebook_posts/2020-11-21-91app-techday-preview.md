下週六 (11/28) 就是 91APP TechDay 的日子了，我在裡面有負責一場 session:

"非同步系統的服務水準保證淺談非同步系統的 SLO 設計"

歡迎有興趣的朋友歡迎來捧場。有現場的席次，也有線上直播~

這場次，我不打算講一些大道理，說監控有多重要等等 blah blah 的，我會直接舉一個例子，說明服務水準這件事應該如何才能達成。

在過去，開發人員很擅長談論 "規格"，上線前也很容易透過測試驗證。但是對於比較難以捉摸的 "服務水準" 就沒這麼容易了，不但量化困難，要測試與驗證也困難。

這次我的 session, 我就會拿我們內部的實際案例當作例子，先從商業目標來訂定期望的 SLO ( Service Level Objective ), 往上對客戶定義 SLA ( Service Level Agreement ), 往下展開需要監控的 SLI ( Service Level Indicator ), 並且依據 SLO 來定義 SLI 在什麼狀況下需要警示。

這一切都已 SLO 為中心，才能展開推進的。因此我非常贊同，從開發的第一天起，就應該把你期待的 SLO 納入考量。另外，當 SLO 不如預期時，調教改善的過程我也會引用 TCO 限制理論的概念 ( 這理論我講的沒有 Ruddy 老師到位... )，如何實際拿來做改善的決策。

當你實作的技術越學越多，明確規格的任務越來越難不倒你的時候，你就會發現，把事情作對 ( do the things right ) 靠的是 skill, 而作對的事 ( do the right thing ) 則是要靠 knowledge .. 不過這轉變，不代表 skill 就不重要了。這個 session 我想表達的就是，如何在你已有的 skills 的基礎上，去驗證 & 發揮你擁有 knowledge 的潛力。

這個 session, 我會同時在兩場研討會上面分享。一個是我們公司自己辦的 91APP TechDay, 另一個是由社群 ( Study4.TW) 舉辦的 .NET 年度盛會 .NET Conf 2020 Taiwan。 不過不用擔心，雖然主題是同一個，但是兩個場子的聽眾屬性是不一樣的，兩次的分享內容會有所差異喔，歡迎大家來參加我的 session, 也歡迎現場找我聊聊 :)
